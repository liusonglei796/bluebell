High-Concurrency Voting Strategy

Handling 10,000+ TPS with Redis & Local Buffering

Current: Direct Lua
  • Every vote triggers a Lua script in Redis.
  • Single-threaded Redis might bottle-neck at 10k+ TPS.
  • Network overhead for every single operation.
Proposed: Local Buffering
  • Aggregate votes in App memory for 100ms.
  • Batch update Redis using Pipeline or Bulk Lua.
  • Reduces Redis CPU usage by ~80%.
A

Redis Cluster / Sharding

Distribute post score ZSets across multiple Redis nodes to spread the load.

B

Local Aggregation (Buffering)

Use a Go worker to batch votes before hitting Redis. Best for "Hot" posts.

C

Adaptive Rate Limiting

Dynamic backoff at the client/API level when Redis latency spikes.
