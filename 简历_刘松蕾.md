### 【1】KamaChat 即时通讯系统 - 高并发架构研究与验证
  
**技术栈** | Go、Gin、MySQL、Redis、Kafka、Docker、Nginx  
**项目规模** | 5 个微服务容器编排

**✓ 容器编排与生产级部署**

- 解决了容器网络通信和 DNS 解析问题，建立了从本地开发 → 生产部署的可重复工程链路
- 配置了服务间依赖关系和健康检查机制，确保服务启动顺序和可用性

### 【2】Bluebell 高并发社区论坛系统

**技术栈** | Go、Gin、GORM、MySQL、Redis、Docker  
**GitHub** | [https://github.com/liusonglei796/bluebell](https://github.com/liusonglei796/bluebell)

#### 项目背景
在系统学习 Go 后端开发时，通过完整构建一个社区论坛系统，掌握了企业级架构设计、高性能优化和工程实践的完整链路。

#### 核心工作

**✓ 企业级架构设计与代码质量**
- 实践了 DDD（领域驱动设计）思想，设计了 **Repository → Service → Handler 三层分离架构**
- 通过**依赖注入（DI）模式**实现代码高度解耦，使单元测试和功能扩展变得容易
- 遵循 **SOLID 原则**（单一职责、开闭原则、里氏替换、接口隔离、依赖倒置）
- 清晰理解了为什么要做分层架构：易于测试、易于维护、易于扩展、技术栈可替换

**✓ 高并发性能优化与并发控制**
- 设计了**多层缓存策略**：Redis 一级缓存 + 本地缓存二级缓存，有效降低 MySQL 热点数据查询压力
- 优化数据库连接池（MaxOpenConns: 200, MaxIdleConns: 30），提升了系统吞吐量
- 实现了**投票幂等性设计**：通过 Redis 原子操作 + 唯一性约束（user_id + post_id），确保高并发投票的数据一致性
- **压测验证结果**：
  - **1491 req/s 吞吐量**（相比单层缓存提升 [X]%）
  - **66ms 平均响应时间**
  - **P99 延迟 89ms**（200 并发下 0% 错误率）

**✓ 完整的企业级工程实践**
- 实现了 **JWT Token 认证**：支持 Access Token + Refresh Token 的双令牌机制
- 设计了实现了简单的管理员权限检查：基于角色的细粒度权限管理
- 实现了**配置热更新**：使用 atomic 原子变量无锁并发更新配置，避免服务重启
- 搭建了 **Docker 多阶段构建 + Compose 编排**的生产级部署方案
  - 多阶段构建减小镜像体积（builder stage 编译 + runtime stage 运行）
  - Compose 编排包含 MySQL、Redis 的自动启动和健康检查
  - 配置了服务依赖关系，确保应用在数据库就绪后才启动
- 集成了 **Swagger API 文档**自动生成
- 应用了 **Zap 结构化日志**框架进行规范的日志管理

**✓ 性能基准测试与验证**
- 使用 go-wrk 进行系统性压测，量化验证了优化效果
- 清晰理解了吞吐量、响应时间、P99/P95 延迟、错误率等性能指标的含义
- 掌握了从开发/测试/生产环境的工程实践差异

#### 技术收获
- 深入理解了 DDD 的核心价值：业务逻辑与技术框架的分离使系统更易演进
- 掌握了缓存穿透、缓存雪崩、缓存击穿等常见问题的解决方案
- 理解了幂等性设计在并发系统中的重要性和实现方式
- 积累了"架构设计 → 性能优化 → 工程验证 → 深度思考"的完整工程能力
