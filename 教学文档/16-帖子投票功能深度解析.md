# 第16章:帖子投票功能深度解析

> **本章导读**
>
> 帖子可以发布和浏览了,但用户如何表达对内容的认可?投票(Vote)功能是社区平台的灵魂机制!
> 本章将深入讲解投票系统的设计思想、业务规则、参数校验。

---

## 1. 投票业务规则设计

### 1.1 核心规则

**规则1: 一人一票**
- 每个用户对每个帖子只能投1票
- 可以投赞成(+1)或反对(-1)，或者取消投票(0)

**规则2: 时间窗口限制**
- 发布后**7天内**可以投票
- 超过7天禁止投票(防止刷旧帖)

## 2. 数据模型设计

### 2.1 投票参数结构体

#### 文件: `internal/dto/request/vote.go`

```go
package request

// VoteRequest 投票参数
type VoteRequest struct {
	// PostID: 目标帖子ID
	PostID int64 `json:"post_id" binding:"required"`

	// Direction: 投票方向
	// 1: 赞成票, 0: 取消投票, -1: 反对票
	Direction int8 `json:"direction" binding:"oneof=1 0 -1"`
}
```

---

## 3. 分数计算逻辑 (Reddit 简化版)

```
帖子分数 = 发布时间戳 + (赞成票 - 反对票) * 432
```

**432 的含义:** 一天有 86400 秒，200 张赞成票可以抵消一天的自然衰减 (86400 / 200 = 432)。

## 4. 投票接口实现

#### 文件: `internal/handler/vote_handler.go`

```go
func (h *Handlers) PostVoteHandler(c *gin.Context) {
    // 1. 参数校验
    p := new(request.VoteRequest)
    if err := c.ShouldBindJSON(p); err != nil {
        backfront.HandleError(c, errorx.ErrInvalidParam)
        return
    }

    // 2. 从 Context 取出经过 Auth 中间件验证的用户ID
    userID, exist := c.Get(CtxUserIDKey)
    if !exist {
        backfront.HandleError(c, errorx.ErrNeedLogin)
        return
    }

    // 3. 调用业务逻辑 (强制类型断言为 int64)
    if err := h.Services.Vote.VoteForPost(c.Request.Context(), userID.(int64), p); err != nil {
        // 直接将 Service 层返回的错误传递给 Handler
        // Service 层使用 errorx.GetCode() 判断具体业务错误
        backfront.HandleError(c, err)
        return
    }

    backfront.ResponseSuccess(c, nil)
}
```

> **错误处理说明**: Handler 层直接将 Service 层返回的错误传递给 `backfront.HandleError`。Service 层使用 `errorx.GetCode()` 判断具体的业务错误（如投票时间已过、重复投票等）。

## 5. 进阶探讨：Redis 与 MySQL 的异步持久化

为了保证在极致性能（2-5ms 响应）的同时，投票数据不因 Redis 故障而永久丢失，我们引入了异步落盘机制。

#### 文件: `internal/service/postsvc/post_service.go`

```go
// VoteForPost 投票业务逻辑
func (s *postServiceStruct) VoteForPost(ctx context.Context, userID int64, p *postreq.VoteRequest) error {
    // 1. 更新 Redis (原子操作，保证实时排名)
    err = s.postCache.VoteForPost(ctx, userIDStr, postIDStr, communityIDStr, float64(p.Direction))
    if err != nil {
        return err
    }

    // 2. 异步落盘到 MySQL (保证数据持久性)
    // 为什么：不阻塞主流程响应，实现最终一致性
    go func() {
        err := s.voteRepo.SaveVote(context.Background(), userID, p.PostID, p.Direction)
        if err != nil {
            zap.L().Error("async save vote to mysql failed", zap.Error(err))
        }
    }()

    return nil
}
```

> **设计思想**: **“Redis 负责高性能计算排名，MySQL 负责高可靠持久化兜底。”** 在面试中，这是一个非常加分的架构设计闭环。
