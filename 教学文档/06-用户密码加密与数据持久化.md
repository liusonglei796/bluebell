## 1. 密码安全:Web 开发的生命线

### 1.1 永远不要存储明文密码

**这是铁律!** 如果数据库中存储的是明文密码(例如 `123456`),一旦发生数据泄露,所有用户账户将瞬间遭受灭顶之灾。

**真实案例:**

2011年,某知名网站遭遇黑客攻击,6000万用户的明文密码被泄露。由于用户通常在多个网站使用相同密码,攻击者利用这些密码进行"撞库"攻击,导致其他平台的账户也被入侵。

**❌ 绝对禁止的做法:**

```sql
-- 明文存储密码 (致命错误!)
INSERT INTO user (username, password)
VALUES ('zhangsan', '123456');
```

**查询结果:**

| user_id | username | password |
|---------|----------|----------|
| 1       | zhangsan | 123456   |

任何有数据库访问权限的人(包括内部员工)都能看到用户密码!

**✅ 正确的做法:**

```sql
-- 存储加密后的密码 (bcrypt 哈希值)
INSERT INTO user (username, password)
VALUES ('zhangsan', '$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy');
```

**查询结果:**

| user_id | username | password |
|---------|----------|----------|
| 1       | zhangsan | $2a$10$N9qo...lhWy |

即使黑客拿到了数据库,也无法直接知道原始密码是 `123456`。

### 1.2 为什么加密不够,必须使用哈希?

很多初学者会混淆"加密"和"哈希"的概念:

| 特性 | 加密(Encryption) | 哈希(Hash) |
|------|-----------------|-----------||
| **可逆性** | ✅ 可解密 | ❌ 不可逆 |
| **密钥** | 需要密钥 | 不需要密钥 |
| **用途** | 保护数据传输 (HTTPS) | 验证数据完整性、密码存储 |
| **安全性** | 如果密钥泄露,数据即泄露 | 即使哈希值泄露,原文也难以还原 |

**为什么密码不能用加密?**

```go
// ❌ 使用 AES 加密存储密码 (错误!)
encryptedPassword := aes.Encrypt("123456", secretKey)
// 如果 secretKey 泄露,所有密码都可以被解密!
```

**正确做法是使用单向哈希:**

```go
// ✅ 使用 bcrypt 哈希 (正确!)
hashedPassword := bcrypt.GenerateFromPassword([]byte("123456"), bcrypt.DefaultCost)
// 即使拿到哈希值,也无法还原出 "123456"
```

## 2. 密码哈希算法对比

### 2.1 常见哈希算法

| 算法 | 输出长度 | 速度 | 安全性 | 推荐用于密码存储 |
|------|---------|------|--------|----------------|
| **MD5** | 128 bit | 极快 | ❌ 已破解 | ❌ 禁用 |
| **SHA-1** | 160 bit | 极快 | ❌ 已破解 | ❌ 禁用 |
| **SHA-256** | 256 bit | 很快 | ⚠️ 可用但不推荐 | ⚠️ 不推荐 |
| **bcrypt** | 184 bit | 慢(可调) | ✅ 安全 | ✅ **强烈推荐** |
| **scrypt** | 可变 | 很慢(可调) | ✅ 安全 | ✅ 推荐 |
| **Argon2** | 可变 | 很慢(可调) | ✅ 最安全 | ✅ 推荐 |

### 2.2 为什么 MD5/SHA-256 不安全?

**问题1: 速度太快**

现代 GPU 每秒可以计算数十亿次 MD5/SHA-256 哈希,暴力破解变得可行。

```bash
# 使用 hashcat 破解 MD5 (实测速度)
# GPU: NVIDIA RTX 4090
# 速度: 200 GH/s (每秒2000亿次哈希计算)
# 8位纯数字密码破解时间: < 1秒
```

**问题2: 彩虹表攻击**

攻击者预先计算常见密码的哈希值,建立"彩虹表":

| 密码 | MD5 哈希值 |
|------|-----------||
| 123456 | e10adc3949ba59abbe56e057f20f883e |
| password | 5f4dcc3b5aa765d61d8327deb882cf99 |
| 111111 | 96e79218965eb72c92a549dd5a330112 |

查表即可反查密码,几乎无成本。

**问题3: 相同密码产生相同哈希**

```go
// MD5 示例
md5("123456") // → e10adc3949ba59abbe56e057f20f883e
md5("123456") // → e10adc3949ba59abbe56e057f20f883e (完全相同!)
```

如果两个用户使用相同密码,哈希值也相同,攻击者可以批量破解。

### 2.3 Bcrypt 的优势

**优势1: 自动加盐(Salt)**

```go
bcrypt.GenerateFromPassword([]byte("123456"), bcrypt.DefaultCost)
// 第一次: $2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
// 第二次: $2a$10$K3eP9jLMNO2pQrS5tUvW7.8fH6gT4yX9vZaB1cD2eF3gH4iJ5kL6m
//         ^^^^^^^^^^^^^^^^^^^^^^^^^^ 这部分是随机盐值
```

**每次加密都会生成不同的盐值**,即使两个用户的密码都是 `123456`,哈希值也完全不同,彩虹表失效!

**优势2: 计算成本可调(Cost Factor)**

```go
// Cost = 10: 约 100ms (默认值,推荐)
bcrypt.GenerateFromPassword(password, 10)

// Cost = 12: 约 400ms (更安全,但慢4倍)
bcrypt.GenerateFromPassword(password, 12)

// Cost = 14: 约 1.6s (极高安全,用于高价值账户)
bcrypt.GenerateFromPassword(password, 14)
```

**"慢即是快"哲学:** 对正常用户来说,登录延迟100ms可以接受,但对攻击者来说,暴力破解的时间成本呈指数级增长。

**破解时间对比:**

| 算法 | Cost | 单次计算时间 | 破解8位密码时间 (GPU) |
|------|------|-------------|---------------------|
| MD5 | - | 0.000001ms | < 1秒 |
| SHA-256 | - | 0.000002ms | < 2秒 |
| bcrypt | 10 | 100ms | **317年** |
| bcrypt | 12 | 400ms | **5072年** |

**优势3: 未来兼容性**

随着硬件性能提升,可以提高 Cost 因子而无需改动代码:

```go
// 2025年: Cost = 10 (默认)
// 2030年: Cost = 12 (配置文件修改即可)
// 无需重构代码,无需重置所有用户密码
```

### 2.4 Bcrypt 哈希格式解析

```
$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
│  │  │  └────────────────────────────────────────────────────────┘
│  │  │                         哈希值 (31字符)
│  │  └───── 盐值 (22字符, Base64编码)
│  └──────── Cost 因子 (10 = 2^10 = 1024次迭代)
└─────────── 算法版本 ($2a$ = bcrypt 版本2a)
```

**关键点:**
- 盐值和哈希值都包含在字符串中,**无需单独存储盐值**
- bcrypt 库会自动解析这个字符串进行密码验证

## 3. Model 层实现:GORM BeforeCreate 钩子

### 3.1 密码加密放在 Model 层

在 DDD 分层架构中,我们选择在 **Model 层** 通过 GORM 的 `BeforeCreate` 钩子自动处理密码加密。

```go
// internal/model/user.go

package model

import (
	"golang.org/x/crypto/bcrypt"
	"gorm.io/gorm"
)

// BcryptCost bcrypt 加密成本参数
// DefaultCost = 10，每增加1，计算时间翻倍
// 性能考虑：生产环境使用 10 可以平衡安全性和性能
const BcryptCost = 10

// User 用户模型
type User struct {
	UserID   int64  `json:"user_id,string" gorm:"column:user_id;primaryKey"`
	Username string `json:"username" gorm:"column:username;uniqueIndex;size:64;not null"`
	Password string `json:"-" gorm:"column:password;size:255;not null"` // json:"-" 防止密码序列化到 JSON
}

// TableName 自定义表名
func (User) TableName() string {
	return "user"
}

// BeforeCreate GORM 钩子，创建前自动处理
// 为什么：将密码加密逻辑放在 Model 层，DAO 层保持纯粹的数据库操作
func (u *User) BeforeCreate(tx *gorm.DB) error {
	if u.Password != "" {
		hash, err := bcrypt.GenerateFromPassword([]byte(u.Password), BcryptCost)
		if err != nil {
			return err
		}
		u.Password = string(hash)
	}
	return nil
}
```

**GORM 的 BeforeCreate 钩子:**

- 在 `db.Create(user)` 执行前自动调用
- 无需在 DAO 层手动调用加密函数
- 保证只要通过 GORM 创建用户,密码一定会被加密

**测试验证:**

```go
package model

import (
	"testing"
	"golang.org/x/crypto/bcrypt"
)

func TestBeforeCreate(t *testing.T) {
	user := &User{Password: "123456"}

	// 模拟 BeforeCreate
	hash, _ := bcrypt.GenerateFromPassword([]byte("123456"), BcryptCost)
	t.Logf("Hash: %s", string(hash))

	// 验证密码
	err := bcrypt.CompareHashAndPassword(hash, []byte("123456"))
	if err != nil {
		t.Error("密码验证失败")
	}
}
```

### 3.2 为什么在 Model 层加密而不是 DAO 层?

**分层职责:**

| 层级 | 职责 | 不应该做的事 |
|------|------|------------|
| **Handler** | 参数校验、响应格式化 | ❌ 不调用数据库 |
| **Service** | 业务流程编排、权限判断 | ❌ 不关心加密方式 |
| **Model** | 数据模型、数据完整性保护 | ✅ **适合放加密逻辑** |
| **DAO** | 数据持久化、SQL 操作 | ❌ 不关心业务细节 |

**方案对比:**

```go
// ❌ 方案A: 在 Service 层加密
func (s *UserService) SignUp(ctx context.Context, p *request.SignUpRequest) error {
    // Service 层需要知道密码如何加密
    encryptedPwd, _ := bcrypt.GenerateFromPassword([]byte(p.Password), 10)
    user := &model.User{Password: string(encryptedPwd)}
    // 问题: 如果未来要更换加密算法, 需要修改 Service 层
}

// ❌ 方案B: 在 DAO 层加密
func (r *UserRepo) InsertUser(ctx context.Context, user *model.User) error {
    user.Password = encryptPassword(user.Password)
    r.db.Create(user)
    // 问题: DAO 层应该只做数据库操作, 不应关注加密
}

// ✅ 方案C: 在 Model 层加密 (当前方案)
func (u *User) BeforeCreate(tx *gorm.DB) error {
    // 利用 GORM 钩子, 自动在插入前加密
    // 优点: 更换加密算法只需改 Model, 不影响其他层
    // 优点: 所有创建用户的路径都会自动加密 (无一遗漏)
}
```

## 4. Domain 层:错误设计模式

### 4.1 Domain 层错误定义

```go
// internal/domain/repository/errors.go

package repository

import "errors"

// Domain 层错误定义
// 将错误提升到 domain 层，使 service 层不再依赖具体的 DAO 包
var (
	ErrUserExist       = errors.New("用户已存在")
	ErrUserNotExist    = errors.New("用户不存在")
	ErrInvalidPassword = errors.New("密码错误")
)
```

**为什么使用 Domain 层预定义错误?**

```go
// ❌ 不好的做法 (旧方式): DAO 层定义错误, Service 层依赖 DAO 包
// dao/mysql/user.go
var ErrorUserExist = errors.New("用户已存在")
// logic/user.go
if err == mysql.ErrorUserExist { ... } // Service 层直接依赖 mysql 包

// ✅ 好的做法 (新方式): Domain 层定义错误, 各层通过接口解耦
// internal/domain/repository/errors.go
var ErrUserExist = errors.New("用户已存在")
// DAO 返回 domain 层错误:
return repository.ErrUserExist
// Service 通过 errors.Is 判断:
if errors.Is(err, repository.ErrUserExist) { ... } // 不依赖具体 DAO 包
```

## 5. DAO 层实现:数据访问

### 5.1 UserRepo 结构体

```go
// internal/dao/mysql/user.go

package mysql

import (

	"context"
	"errors"
	"fmt"

	"golang.org/x/crypto/bcrypt"
	"gorm.io/gorm"

// UserRepo 用户数据访问实现
type UserRepo struct {
	db *gorm.DB
}

// NewUserRepo 创建 UserRepo 实例
func NewUserRepo(db *gorm.DB) *UserRepo {
	return &UserRepo{db: db}
}
```

### 5.2 检查用户是否存在

```go
// CheckUserExist 检查指定用户名的用户是否存在
func (r *UserRepo) CheckUserExist(ctx context.Context, username string) (err error) {
	var count int64
	// GORM 使用 Count 方法统计记录数
	err = r.db.WithContext(ctx).Model(&model.User{}).Where("username = ?", username).Count(&count).Error
	if err != nil {
		return err
	}
	if count > 0 {
		return repository.ErrUserExist // 返回 Domain 层定义的错误
	}
	return nil
}
```

**为什么用 COUNT 而不是直接查询用户?**

```go
// ❌ 方法1: 直接查询用户 (浪费资源)
var user model.User
err := r.db.Where("username = ?", username).First(&user).Error
// 缺点: 查询了不需要的字段,浪费网络带宽和内存

// ✅ 方法2: 只查询数量 (推荐)
var count int64
err := r.db.Model(&model.User{}).Where("username = ?", username).Count(&count).Error
// 优点: 只查询数量,效率高
```

### 5.3 插入用户数据

```go
// InsertUser 插入新用户
// 密码加密已移至 Model 的 BeforeCreate 钩子中自动处理
func (r *UserRepo) InsertUser(ctx context.Context, user *model.User) (err error) {
	// GORM Create 会在执行前自动调用 BeforeCreate 钩子
	// → BeforeCreate 中会自动加密密码
	err = r.db.WithContext(ctx).Create(user).Error
	if err != nil {
		return fmt.Errorf("插入用户失败: %w", err)
	}
	return nil
}
```

**注意:** 不再需要手动调用 `encryptPassword()`,GORM 的钩子机制自动处理。

## 6. Service 层实现:业务流程编排

### 6.1 注册业务流程

```go
// internal/service/user/user_service.go

package user

import (
	"bluebell/internal/domain/repository"
	"bluebell/internal/dto/request"
	"bluebell/internal/model"
	"bluebell/pkg/errorx"
	"bluebell/pkg/snowflake"
	"context"
	"errors"

	"go.uber.org/zap"
)

// UserService 用户业务逻辑服务
type UserService struct {
	userRepo   repository.UserRepository
	tokenCache repository.UserTokenCacheRepository
}

// NewUserService 创建用户服务实例
func NewUserService(userRepo repository.UserRepository, tokenCache repository.UserTokenCacheRepository) *UserService {
	return &UserService{
		userRepo:   userRepo,
		tokenCache: tokenCache,
	}
}

// SignUp 处理用户注册业务逻辑
func (s *UserService) SignUp(ctx context.Context, p *request.SignUpRequest) (err error) {
	// 1️⃣ 判断用户是否存在
	if err = s.userRepo.CheckUserExist(ctx, p.Username); err != nil {
		if errors.Is(err, repository.ErrUserExist) {
			return errorx.ErrUserExist
		}
		zap.L().Error("userRepo.CheckUserExist failed",
			zap.String("username", p.Username),
			zap.Error(err))
		return errorx.ErrServerBusy
	}

	// 2️⃣ 生成 UserID (雪花算法)
	userID := snowflake.GetID()

	// 3️⃣ 构造 User 实例
	// 注意: 密码此时还是明文,进入 DAO 层后
	// GORM BeforeCreate 钩子会自动加密
	u := &model.User{
		UserID:   userID,
		Username: p.Username,
		Password: p.Password,
	}

	// 4️⃣ 保存到数据库
	err = s.userRepo.InsertUser(ctx, u)
	if err != nil {
		zap.L().Error("userRepo.InsertUser failed",
			zap.Int64("user_id", userID),
			zap.String("username", p.Username),
			zap.Error(err))
		return errorx.ErrServerBusy
	}

	return nil
}
```

## 7. Handler 层完整实现

```go
// internal/handler/user_handler.go

func (h *Handlers) SignUpHandler(c *gin.Context) {
	// 1️⃣ 参数绑定和校验
	p := new(request.SignUpRequest)
	if err := c.ShouldBindJSON(p); err != nil {
		errs, ok := err.(validator.ValidationErrors)
		if !ok {
			HandleError(c, errorx.ErrInvalidParam)
			return
		}
		ResponseErrorWithMsg(c, errorx.CodeInvalidParam, removeTopStruct(errs.Translate(trans)))
		return
	}

	// 2️⃣ 调用业务逻辑 (通过依赖注入)
	if err := h.Services.User.SignUp(c.Request.Context(), p); err != nil {
		// 3️⃣ 统一错误处理
		// HandleError 自动识别 errorx.CodeError 并返回对应的错误码
		HandleError(c, err)
		return
	}

	// 4️⃣ 返回成功响应
	ResponseSuccess(c, nil)
}
```

## 10. 常见问题 FAQ

### Q1: bcrypt 的 Cost 因子应该设置为多少?

**A:**

| Cost | 计算时间 | 适用场景 |
|------|---------|---------||
| 4-6 | < 10ms | ❌ 测试环境,不安全 |
| 10 | ~100ms | ✅ **推荐** (默认值) |
| 12 | ~400ms | 高价值账户 (管理员、付费用户) |
| 14+ | > 1s | 极高安全需求 (银行、政府) |

---

### Q2: 为什么不用 MD5 + 盐值?

**A:**

即使加盐,MD5 仍然不安全:

```go
// ❌ MD5 + 盐值 (不推荐)
salt := "random_salt_12345"
hash := md5(password + salt)
// 问题: MD5 太快,GPU 每秒可计算数十亿次

// ✅ bcrypt (推荐)
hash := bcrypt.GenerateFromPassword(password, 10)
// 优势: 慢,且自动加盐
```

---

### Q3: 如何验证密码?

**A:**

```go
// internal/dao/mysql/user.go

// CheckLogin 登录验证
func (r *UserRepo) CheckLogin(ctx context.Context, user *model.User) (err error) {
	oPassword := user.Password // 用户输入的明文密码

	// 查询数据库,获取用户信息(包含加密后的密码)
	err = r.db.WithContext(ctx).Where("username = ?", user.Username).First(user).Error
	if err != nil {
		if errors.Is(err, gorm.ErrRecordNotFound) {
			return repository.ErrUserNotExist
		}
		return fmt.Errorf("login failed: %w", err)
	}

	// 使用 bcrypt 验证密码
	err = bcrypt.CompareHashAndPassword([]byte(user.Password), []byte(oPassword))
	if err != nil {
		return repository.ErrInvalidPassword
	}
	return nil
}
```

---

### Q4: 如果要更换加密算法怎么办?

**A:**

**方案1: 渐进式迁移 (推荐)**

由于密码加密集中在 Model 的 `BeforeCreate` 钩子中,更换算法只需修改一个地方:

```go
// internal/model/user.go

func (u *User) BeforeCreate(tx *gorm.DB) error {
	if u.Password != "" {
		// 只需在这里更换加密算法
		hash, err := argon2.Hash(u.Password) // 换成 Argon2
		if err != nil {
			return err
		}
		u.Password = hash
	}
	return nil
}
```

登录验证时可以做版本兼容:

```go
func (r *UserRepo) CheckLogin(ctx context.Context, user *model.User) error {
	// ... 查询用户 ...
	if strings.HasPrefix(user.Password, "$2a$") {
		// bcrypt 格式
		return bcrypt.CompareHashAndPassword(...)
	} else if strings.HasPrefix(user.Password, "$argon2") {
		// Argon2 格式
		return argon2.Compare(...)
	}
}
```

### Q5: 为什么使用 GORM 而不是原生 SQL?

**A:**

**GORM 的优势:**

1. **面向对象**: 直接操作 Struct,自动处理字段映射
   ```go
   // ✅ 简单直观
   r.db.WithContext(ctx).Create(user)
   ```

2. **安全性**: 自动防止 SQL 注入

3. **钩子机制**: `BeforeCreate`、`AfterCreate` 等钩子可以自动处理密码加密等逻辑

4. **可维护性**: 字段变更时无需修改大量 SQL,只需修改 Struct Tag

注意: 在极高性能要求的场景下,可以考虑使用原生 SQL 或 `gorm.Raw()`。

## 11. 最佳实践总结

### 11.1 安全清单

- ✅ 使用 bcrypt 存储密码 (Cost ≥ 10)
- ✅ 绝不明文存储密码
- ✅ 绝不在日志中记录明文密码
- ✅ 密码传输使用 HTTPS
- ✅ 定期审计密码策略 (长度、复杂度)
- ✅ 检测弱密码 (123456, password 等)

### 11.2 代码规范

- ✅ 密码加密在 Model 层 `BeforeCreate` 钩子中实现
- ✅ 使用 Domain 层预定义错误 (`repository.ErrUserExist`)
- ✅ Service 层通过 Repository 接口访问数据
- ✅ 记录详细日志 (但不记录敏感信息)
- ✅ 使用 `ctx context.Context` 支持超时控制

### 11.3 性能优化

- ✅ 使用 GORM 连接池 (默认开启)
- ✅ 用 COUNT 检查存在性,不查询完整行
- ✅ 为 username 字段创建唯一索引
- ✅ 批量插入使用事务
