# 第10章:单点登录与互踢模式

> **本章导读**
>
> 什么是**单点登录 (SSO - Single Sign On)**?在广义上它指"一处登录,处处访问"。但在移动互联网时代,它常指**"单设备登录"**,即同一账号同一时间只能在一个设备在线。当你在手机 B 上登录时,手机 A 会自动下线(被踢)。
>
> 传统的 Session 模式天然支持这一点(服务器删除旧 Session 即可),但无状态的 JWT 该如何实现?本章将结合 Redis 实现这一功能,并深入探讨多设备管理、安全隐患和最佳实践。

## 1. 核心原理深度剖析

### 1.1 无状态 JWT 的困境

JWT 的核心优势是**无状态**,但这也带来了一个致命问题:

```
用户在设备 A 登录 → 获得 Token_A (有效期 10 分钟)
  ↓
用户在设备 B 登录 → 获得 Token_B (有效期 10 分钟)
  ↓
问题: Token_A 和 Token_B 都有效!
  ↓
服务器无法识别 Token_A 已经"过时"
  ↓
用户可以在两个设备上同时操作 (安全风险)
```

**传统 Session 的解决方案:**

```go
// Session 模式 (天然支持互踢)
func Login(userID int64) sessionID string {
    // 1. 删除该用户的旧 Session
    sessionStore.Delete("session:" + userID)

    // 2. 创建新 Session
    newSessionID := uuid.New()
    sessionStore.Set("session:" + userID, newSessionID)

    return newSessionID
}
```

**JWT 的困境:**

```go
// JWT 模式 (无法互踢)
func Login(userID int64) token string {
    // 生成新 Token
    newToken := jwt.Sign(userID, secret)

    // 问题: 旧 Token 依然有效!
    // 服务器没有存储任何状态,无法知道旧 Token 的存在

    return newToken
}
```

### 1.2 Redis 解决方案

我们引入 Redis 作为**权威的状态中心**,将 JWT 的"无状态"转变为"轻状态"。

**核心规则:**

1. **Redis 是唯一的真相来源**: Redis 中只保存该用户**当前唯一有效**的 Access Token。
2. **登录即覆盖**: 生成新 Token 时,写入 Redis (`SET user:token:{uid} new_token`),覆盖旧值。
3. **双重校验**: 中间件不仅验证 JWT 签名,还对比 `Request.Token == Redis.Token`。

**架构示意图:**

```
┌─────────────┐
│   客户端    │
│ (设备 A/B)  │
└──────┬──────┘
       │ HTTP Request
       │ Header: Authorization: Bearer {token}
       ↓
┌─────────────┐
│ JWT 中间件  │
│             │
│ 1. 验证签名 ├────────────┐
│ 2. 查询Redis│            │
└──────┬──────┘            │
       │                   │
       │ Get token         │
       ↓                   ↓
┌─────────────┐      ┌──────────┐
│    Redis    │      │   JWT    │
│             │      │  Verify  │
│ Key: user:  │      │          │
│ token:{uid} │      └──────────┘
│             │
│ Value:      │
│ {current_   │
│  token}     │
└─────────────┘

校验逻辑:
if Request.Token == Redis.Token:
    ✅ 通过 (最新 Token)
else:
    ❌ 拒绝 (旧 Token,已被踢下线)
```

**工作流程:**

```
【设备 A 登录】
1. 用户在设备 A 输入账号密码
2. 服务器验证通过,生成 Token_A
3. Redis.SET("user:token:1001", Token_A, TTL=10min)
4. 返回 Token_A 给设备 A

【设备 A 访问接口】
5. 设备 A 携带 Token_A 请求接口
6. 中间件验证: Token_A == Redis.GET("user:token:1001")
7. ✅ 通过,正常返回数据

【设备 B 登录 (踢掉设备 A)】
8. 用户在设备 B 输入账号密码
9. 服务器验证通过,生成 Token_B
10. Redis.SET("user:token:1001", Token_B, TTL=10min)  // 覆盖 Token_A
11. 返回 Token_B 给设备 B

【设备 A 再次访问接口 (被踢)】
12. 设备 A 携带 Token_A 请求接口
13. 中间件验证: Token_A != Redis.GET("user:token:1001")  // 不一致!
14. ❌ 拒绝,返回 "账号已在其他设备登录"
15. 设备 A 跳转登录页
```

### 1.3 为什么不在 JWT Claims 中加设备ID?

**错误方案 (❌):**

```go
type UserClaims struct {
    UserID   int64  `json:"user_id"`
    DeviceID string `json:"device_id"`  // 设备标识
    jwt.RegisteredClaims
}

// 中间件中检查
if claims.DeviceID != currentDeviceID {
    return errors.New("设备不匹配")
}
```

**问题:**

1. **无法踢人**: 旧设备的 Token 依然有效,只是在"另一台设备"上无法使用。
2. **绕过风险**: 攻击者窃取 Token 后,只要伪造设备ID就能使用。
3. **管理复杂**: 无法实现"查看所有在线设备"、"踢掉指定设备"等功能。

**正确方案 (✅): Redis 状态管理**

- Redis 是唯一的真相来源,Token 无需感知设备信息。
- 服务器完全控制 Token 的有效性,安全可控。

## 2. Redis 层实现

### 2.1 Key 设计模式

Redis Key 的设计直接影响性能和可维护性。

#### 方案对比

| 方案 | Key 格式 | 优点 | 缺点 |
|------|---------|------|------|
| ❌ 平铺 | `user_token_1001` | 简单 | 难以批量操作,命名空间污染 |
| ✅ 分层 | `bluebell:user:token:1001` | 清晰分组,支持通配查询 | Key 稍长 |
| ✅ Hash | `bluebell:user:1001` (Hash) | 节省内存 | 无法单独设置过期时间 |

**我们的选择: 分层 String**

```
Key: bluebell:active_access_token:{userID}
Value: {access_token}
TTL: 10 分钟 (与 Access Token 过期时间一致)

Key: bluebell:active_refresh_token:{userID}
Value: {refresh_token}
TTL: 30 天 (与 Refresh Token 过期时间一致)
```

**为什么用 String 而不是 Hash?**

- String 可以单独设置 TTL,自动过期。
- Hash 需要手动清理,容易遗漏。
- 性能差异可忽略(单用户只有 2 个 Key)。

在 `internal/dao/cache/user/user.go` 中:

```go
package usercache

// Redis Keys 相关常量
const (
	keyPrefix           = "bluebell:"
	keyUserAccessToken  = "active_access_token:"  // bluebell:active_access_token:1001
	keyUserRefreshToken = "active_refresh_token:" // bluebell:active_refresh_token:1001
)

// getRedisKey 生成完整的 Redis Key
func getRedisKey(key string) string {
	return keyPrefix + key
}
```

**设计原则:**

1. **项目前缀**: 避免与其他项目的 Key 冲突(特别是共享 Redis 实例时)。
2. **分层命名**: `项目前缀:具体功能`,语义清晰。
3. **统一函数**: `getRedisKey()` 确保所有 Key 都经过前缀处理。

### 2.3 存取 Token 方法

在 `internal/dao/cache/user/user.go` 中:

```go
// SetUserToken 存储用户的 Access Token 和 Refresh Token
func (c *userTokenCacheStruct) SetUserToken(ctx context.Context, userID int64, aToken, rToken string, aExp, rExp time.Duration) error {
	pipe := c.rdb.Pipeline()

	pipe.Set(ctx, getRedisKey(keyUserAccessToken+fmt.Sprint(userID)), aToken, aExp)
	pipe.Set(ctx, getRedisKey(keyUserRefreshToken+fmt.Sprint(userID)), rToken, rExp)

	_, err := pipe.Exec(ctx)
	if err != nil {
		return errorx.Wrapf(err, errorx.CodeCacheError, "set user token pipeline exec failed (user_id: %d)", userID)
	}
	return nil
}

// GetUserAccessToken 获取当前有效的 Access Token
func (c *userTokenCacheStruct) GetUserAccessToken(ctx context.Context, userID int64) (string, error) {
	return c.rdb.Get(ctx, getRedisKey(keyUserAccessToken+fmt.Sprint(userID))).Result()
}

// DeleteUserToken 删除用户的 Token (用于登出/修改密码)
func (c *userTokenCacheStruct) DeleteUserToken(ctx context.Context, userID int64) error {
	pipe := c.rdb.Pipeline()
	pipe.Del(ctx, getRedisKey(keyUserAccessToken+fmt.Sprint(userID)))
	pipe.Del(ctx, getRedisKey(keyUserRefreshToken+fmt.Sprint(userID)))
	_, err := pipe.Exec(ctx)
	return err
}
```

**为什么用 Pipeline?**

```go
// ❌ 不使用 Pipeline (2 次网络往返)
rdb.Set(ctx, keyAT, aToken, aExp)  // RTT 1
rdb.Set(ctx, keyRT, rToken, rExp)  // RTT 2

// ✅ 使用 Pipeline (1 次网络往返)
pipe := rdb.Pipeline()
pipe.Set(ctx, keyAT, aToken, aExp)
pipe.Set(ctx, keyRT, rToken, rExp)
pipe.Exec(ctx)  // 一次性发送
```

**性能对比:**

- 不使用 Pipeline: 2ms (网络延迟) × 2 = 4ms
- 使用 Pipeline: 2ms (网络延迟) × 1 = 2ms
- **性能提升: 50%**

## 3. 业务逻辑层集成

### 3.1 登录逻辑改造

在 `internal/service/usersvc/user_service.go` 中,生成 Token 后立即存入 Redis:

```go
// Login 处理用户登录逻辑
func (s *userServiceStruct) Login(ctx context.Context, p *userreq.LoginRequest) (resp *userresp.LoginResponse, err error) {
	// ... 验证用户名和密码
	
	// 2. 生成 JWT Token
	aToken, rToken, err := jwt.GenToken(s.cfg, user.UserID)
	if err != nil {
		return nil, err
	}

	// 3. 【核心】存入 Redis,实现互踢
	// 为什么: 这会覆盖掉该用户之前在 Redis 中的 Token
	//         旧设备的 Token 会在中间件中被拒绝 (与 Redis 不匹配)
	err = s.tokenRepo.SetUserToken(
		ctx,
		user.UserID,
		aToken,
		rToken,
		s.cfg.JWT.AccessTokenExpire,
		s.cfg.JWT.RefreshTokenExpire,
	)
	if err != nil {
		return nil, err
	}

	return &userresp.LoginResponse{
		AccessToken:  aToken,
		RefreshToken: rToken,
	}, nil
}
```

**逻辑流程:**

```
用户登录
    ↓
1. 验证密码 (MySQL)
    ├─ 用户不存在 → 返回错误
    ├─ 密码错误 → 返回错误
    └─ 验证通过 → 继续
    ↓
2. jwt.GenToken() ─────  生成双 Token
    ├─ Access Token
    └─ Refresh Token
    ↓
3. tokenRepo.SetUserToken() ─ 存储到 Redis
    ├─ SET active_access_token:{id} = {aToken}
    └─ SET active_refresh_token:{id} = {rToken}
    ↓
返回双 Token 给客户端
```

### 3.2 刷新逻辑改造

在 `internal/service/usersvc/user_service.go` 中,刷新 Token 后也必须更新 Redis:

```go
// RefreshToken 刷新 Access Token
func (s *userServiceStruct) RefreshToken(ctx context.Context, p *userreq.RefreshTokenRequest) (resp *userresp.LoginResponse, err error) {
	// 1. 验证并解析 Refresh Token
	userID, err := jwt.ParseToken(s.cfg, p.RefreshToken)
	if err != nil {
		return nil, errorx.ErrInvalidToken
	}

	// 2. 生成新 Token
	newAToken, newRToken, err := jwt.GenToken(s.cfg, userID)
	if err != nil {
		return nil, err
	}

	// 3. 【关键】更新 Redis 状态
	// 为什么: 如果不更新,新生成的 Token 将无法通过中间件校验
	err = s.tokenRepo.SetUserToken(
		ctx,
		userID,
		newAToken,
		newRToken,
		s.cfg.JWT.AccessTokenExpire,
		s.cfg.JWT.RefreshTokenExpire,
	)
	if err != nil {
		return nil, err
	}

	return &userresp.LoginResponse{
		AccessToken:  newAToken,
		RefreshToken: newRToken,
	}, nil
}
```

**为什么刷新也要更新 Redis?**

```
场景: 用户的 Access Token 过期,前端自动刷新

旧 Access Token: Token_A (过期)
旧 Refresh Token: RToken_A

前端调用刷新接口:
  ↓
服务器生成新 Token:
  新 Access Token: Token_B
  新 Refresh Token: RToken_B
  ↓
如果不更新 Redis:
  Redis 中仍是 Token_A
  ↓
前端用 Token_B 请求接口:
  中间件对比: Token_B != Redis(Token_A)
  ↓
❌ 拒绝! (明明是合法的新 Token,却被拒绝)

如果更新 Redis:
  Redis 更新为 Token_B
  ↓
前端用 Token_B 请求接口:
  中间件对比: Token_B == Redis(Token_B)
  ↓
✅ 通过!
```

```go
	}

	return nil
}
```

## 4. 中间件实现互踢校验

在 `internal/middleware/auth.go` 中,这是实现"踢人"逻辑的核心执行点:

```go
package middleware

import (
	"bluebell/internal/backfront"
	"bluebell/internal/config"
	"bluebell/internal/domain/cachedomain"
	"bluebell/internal/infrastructure/jwt"
	"bluebell/pkg/errorx"
	"strings"

	"github.com/gin-gonic/gin"
)

// JWTAuthMiddleware 基于JWT的认证中间件，包含 SSO 校验
func JWTAuthMiddleware(cfg *config.Config, tokenRepo cachedomain.UserTokenCacheRepository) gin.HandlerFunc {
	return func(c *gin.Context) {
		// 1. 获取 Authorization Header
		authHeader := c.Request.Header.Get("Authorization")
		if authHeader == "" {
			backfront.HandleError(c, errorx.ErrNeedLogin)
			c.Abort()
			return
		}

		// 2. 解析 Bearer Token
		parts := strings.SplitN(authHeader, " ", 2)
		if !(len(parts) == 2 && parts[0] == "Bearer") {
			backfront.HandleError(c, errorx.ErrInvalidToken)
			c.Abort()
			return
		}
		tokenStr := parts[1]

		// 3. 解析并校验 aToken
		userID, err := jwt.ParseToken(cfg, tokenStr)
		if err != nil {
			backfront.HandleError(c, errorx.ErrInvalidToken)
			c.Abort()
			return
		}

		// 4. SSO 校验 (Redis 实时校验)
		activeToken, err := tokenRepo.GetUserAccessToken(c.Request.Context(), userID)
		if err != nil {
			// Redis 异常时降级处理：仅依赖 JWT 自身校验结果
		} else {
			if activeToken != tokenStr {
				backfront.HandleError(c, errorx.New(errorx.CodeInvalidToken, "账号已在其他地方登录"))
				c.Abort()
				return
			}
		}

		// 5. 将用户信息存入上下文
		c.Set("UserIDKey", userID)
		c.Next()
	}
}
```

**校验流程图:**

```
HTTP 请求到达
    ↓
提取 Authorization Header
    ├─ 没有 → 401 需要登录
    └─ 有 → 继续
    ↓
解析 "Bearer <token>"
    ├─ 格式错误 → 401 Token 格式错误
    └─ 格式正确 → clientToken
    ↓
JWT 签名验证
    ├─ 签名错误 → 401 Token 无效
    ├─ Token 过期 → 401 Token 无效
    └─ 验证通过 → UserClaims (UserID)
    ↓
查询 Redis
    ├─ Key 不存在 → 401 需要登录 (Token 已失效)
    └─ 查询成功 → serverToken
    ↓
对比 Token
    ├─ clientToken != serverToken → 401 账号已在其他设备登录
    └─ clientToken == serverToken → ✅ 通过
    ↓
注入 UserID 到上下文
    ↓
放行 (c.Next)
    ↓
业务 Handler 执行
```

## 5. 完整场景推演

### 5.1 场景一: 正常登录和访问

```
【时间线】
T0: 用户在手机上登录

1. 客户端: POST /login {username: "lay", password: "123456"}
2. 服务器:
   - 验证密码 ✅
   - 生成 Token_Phone
   - Redis.SET("bluebell:user:access_token:1001", Token_Phone, 10min)
   - 返回 Token_Phone

T1: 用户访问帖子列表 (第 1 分钟)

3. 客户端: GET /posts, Header: Bearer Token_Phone
4. 中间件:
   - JWT 签名验证 ✅
   - Redis.GET("bluebell:user:access_token:1001") = Token_Phone
   - Token_Phone == Token_Phone ✅
   - 放行
5. 服务器: 返回帖子列表

T5: 用户继续访问 (第 5 分钟)

6. 客户端: GET /community, Header: Bearer Token_Phone
7. 中间件: 同上,验证通过 ✅
8. 服务器: 返回社区列表
```

```
【时间线】
T0: 用户在手机上登录 (继续上面的场景)
Redis: Token_Phone

T6: 用户在电脑上登录 (第 6 分钟)

   - 放行
9. 服务器: 返回帖子列表
```

### 5.3 场景三: 修改密码强制下线

```
【时间线】
T0: 用户在多个设备登录
  - 手机: Token_Phone
  - 电脑: Token_PC (最新)
  - 平板: Token_Pad (旧,已被踢)

Redis: Token_PC

T1: 用户在网页上修改密码

1. 客户端 (PC): POST /change_password
   {
     "old_password": "123456",
     "new_password": "newpass"
   }
2. 服务器:
   - 验证旧密码 ✅
   - 更新密码到数据库
   - 【关键】redis.DeleteUserToken(userID)  // 删除 Redis 中的 Token
3. 返回成功

Redis 状态变化:
  旧值: Token_PC
  新值: (Key 被删除)

T2: 用户在任意设备访问接口

4. 客户端: GET /posts, Header: Bearer Token_PC (或任何旧 Token)
5. 中间件:
   - JWT 签名验证 ✅ (Token 未过期)
   - Redis.GET("bluebell:user:access_token:1001") = nil (Key 不存在)
   - ❌ 拒绝!
6. 服务器: 返回 401 "需要登录"
7. 客户端: 跳转登录页

用户必须重新登录:
8. 输入新密码
9. 生成新 Token
10. 写入 Redis
11. 正常使用
```

## 6. 多设备管理(扩展功能)

目前的方案只支持"互踢模式"(单设备登录)。如果要支持多设备并存,需要改进 Redis Key 设计。

### 6.1 多设备共存方案

**改进的 Redis Key 设计:**

```
原方案 (单设备):
  Key: bluebell:user:access_token:{userID}
  Value: {token}

改进方案 (多设备):
  Key: bluebell:user:access_token:{userID}:{deviceID}
  Value: {token}
```

**设备ID 生成 (前端):**

```javascript
// 浏览器环境
const deviceID = localStorage.getItem('device_id') || generateDeviceID()

function generateDeviceID() {
  const id = `${navigator.userAgent}_${screen.width}x${screen.height}_${Date.now()}`
  const hash = SHA256(id)  // 简化为哈希值
  localStorage.setItem('device_id', hash)
  return hash
}
```

**后端改造:**

```go
// 登录时传入 deviceID
type ParamLogin struct {
    Username string `json:"username" binding:"required"`
    Password string `json:"password" binding:"required"`
    DeviceID string `json:"device_id" binding:"required"`  // 新增
}

// Redis 存储改造
func SetUserTokenByDevice(userID int64, deviceID string, aToken, rToken string, aExp, rExp time.Duration) error {
    pipe := rdb.Pipeline()

    keyAT := fmt.Sprintf("%suser:access_token:%d:%s", KeyPrefix, userID, deviceID)
    pipe.Set(ctx, keyAT, aToken, aExp)

    keyRT := fmt.Sprintf("%suser:refresh_token:%d:%s", KeyPrefix, userID, deviceID)
    pipe.Set(ctx, keyRT, rToken, rExp)

    _, err := pipe.Exec(ctx)
    return err
}
```

**设备列表查询:**

```go
// 查询用户所有在线设备
func GetUserDevices(userID int64) ([]string, error) {
    pattern := fmt.Sprintf("%suser:access_token:%d:*", KeyPrefix, userID)
    keys, err := rdb.Keys(ctx, pattern).Result()
    if err != nil {
        return nil, err
    }

    devices := make([]string, 0, len(keys))
    for _, key := range keys {
        // 提取 deviceID
        parts := strings.Split(key, ":")
        deviceID := parts[len(parts)-1]
        devices = append(devices, deviceID)
    }

    return devices, nil
}
```

## 7. 安全性分析

### 7.1 并发登录问题

**场景:** 用户在极短时间内(毫秒级)在两个设备上同时登录。

```
T0: 设备 A 和设备 B 同时发起登录请求

设备 A:
  1. 验证密码 ✅
  2. 生成 Token_A
  3. Redis.SET("user:token:1001", Token_A)
  4. 返回 Token_A

设备 B:
  1. 验证密码 ✅
  2. 生成 Token_B
  3. Redis.SET("user:token:1001", Token_B)  // 覆盖 Token_A
  4. 返回 Token_B

结果: Token_A 立即失效,设备 A 被踢下线
```

**这是预期行为还是 Bug?**

- **预期行为**: 互踢模式下,最后登录的设备生效。
- **如果不希望互踢**: 改用多设备共存方案。

### 7.2 Redis 故障降级

**问题:** 如果 Redis 宕机,所有用户都无法访问接口。

**降级方案:**

```go
func JWTAuthMiddleware() func(c *gin.Context) {
    return func(c *gin.Context) {
        // ... JWT 签名验证 ...

        // Redis 校验
        serverToken, err := redis.GetUserAccessToken(mc.UserID)
        if err != nil {
            // 判断是否是 Redis 连接错误
            if errors.Is(err, redis.ErrRedisDown) {
                // 降级: 跳过 Redis 验证,仅依赖 JWT 签名
                zap.L().Warn("Redis is down, fallback to JWT-only validation")
                c.Set(controller.CtxUserIDKey, mc.UserID)
                c.Next()
                return
            }

            // 其他错误 (如 Key 不存在),正常拒绝
            controller.ResponseError(c, controller.CodeNeedLogin)
            c.Abort()
            return
        }

        // 正常校验
        if clientToken != serverToken {
            controller.ResponseErrorWithMsg(c, controller.CodeInvalidToken, "账号已在其他设备登录")
            c.Abort()
            return
        }

        c.Set(controller.CtxUserIDKey, mc.UserID)
        c.Next()
    }
}
```

**权衡:**

- ✅ 降级后服务可用性提升
- ❌ 失去互踢功能,存在安全风险

## 8. 最佳实践总结

1. ✅ **Redis Key 设计**: 使用分层命名,统一前缀,方便管理。
2. ✅ **Pipeline 优化**: 批量操作减少网络往返,提升性能。
3. ✅ **TTL 自动过期**: 利用 Redis TTL 自动清理过期 Key,无需手动维护。
4. ✅ **双重校验**: JWT 签名 + Redis 状态,安全性更高。
5. ✅ **降级方案**: Redis 故障时,可降级为纯 JWT 模式(可选)。
6. ✅ **日志记录**: 记录互踢事件,方便排查异常登录。
