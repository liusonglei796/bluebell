## 1. 为什么需要参数校验?

### 1.1 用户输入不可信

**永远不要相信用户输入!** 这是 Web 开发的黄金法则。

```go
// ❌ 危险示例:不做任何校验
func SignUpHandler(c *gin.Context) {
    var p request.SignUpRequest
    c.ShouldBindJSON(&p)
    // 直接使用 p.Username 和 p.Password,可能为空字符串!
    logic.SignUp(&p) // 💥 可能导致数据库插入空值或异常
}
```

**可能的问题:**
- 用户名为空或超长
- 密码强度不足
- 邮箱格式不正确
- SQL 注入风险
- 恶意数据导致程序崩溃

### 1.2 校验的层次

在 Bluebell 项目中,我们采用多层校验策略:

| 校验层 | 位置 | 职责 | 示例 |
|--------|------|------|------|
| **前端校验** | Vue.js 表单 | 用户体验优化,即时反馈 | 输入时提示"用户名不能为空" |
| **参数绑定校验** | Controller 层 | 格式和必填项校验 | binding:"required" |
| **业务逻辑校验** | Logic 层 | 业务规则校验 | 检查用户名是否已存在 |
| **数据库约束** | MySQL | 最后的防线 | UNIQUE KEY, NOT NULL |

**原则:** 前端校验提升用户体验,后端校验保障安全性。后端必须做完整校验,即使前端已经校验过。

## 2. Gin 参数绑定机制

### 2.1 绑定方法对比

Gin 提供了多种参数绑定方法:

| 方法 | 特点 | 使用场景 |
|------|------|---------|
| **ShouldBindJSON** | 从 JSON 请求体绑定,不会终止请求 | ✅ 推荐用于 API 接口 |
| **BindJSON** | 从 JSON 请求体绑定,校验失败自动返回 400 | 简单场景,但无法自定义错误响应 |
| **ShouldBind** | 自动识别 Content-Type (JSON/Form/Query) | 兼容多种请求类型 |
| **ShouldBindQuery** | 从 URL 查询参数绑定 | GET 请求参数 |
| **ShouldBindUri** | 从路径参数绑定 | RESTful 风格路径参数 |

**我们选择 `ShouldBindJSON` 的原因:**

```go
// ✅ ShouldBindJSON:校验失败不会自动响应,可以自定义错误格式
if err := c.ShouldBindJSON(&p); err != nil {
    // 自定义错误响应,统一 API 风格
    ResponseError(c, CodeInvalidParam)
    return
}

// ❌ BindJSON:校验失败自动返回 400,无法统一错误格式
if err := c.BindJSON(&p); err != nil {
    // 这里永远不会执行,因为 BindJSON 已经自动响应了
    return
}
```

### 2.2 参数结构体定义

在 Bluebell 项目中,我们在 `models/params.go` 中定义所有 API 参数结构体:

```go
// internal/dto/request/user.go

// SignUpRequest 注册请求参数
// 为什么:定义前端传递的 JSON 数据结构,使用 binding 标签进行参数校验
type SignUpRequest struct {
    // binding:"required" 表示该字段必填,如果为空 Gin 会报错
    Username string `json:"username" binding:"required"`
    Password string `json:"password" binding:"required"`
    // re_password 用于确认密码,业务逻辑中会校验两次密码是否一致
    RePassword string `json:"re_password" binding:"required"`
}
```

**关键点解析:**

1. **json 标签**: 指定 JSON 字段名,前端传 `"username"` 会绑定到 `Username` 字段
2. **binding 标签**: 定义校验规则,支持 Validator v10 的所有标签
3. **字段命名**: Go 结构体使用大驼峰(PascalCase),JSON 使用蛇形命名(snake_case)

**前端请求示例:**

```json
POST /api/v1/signup
Content-Type: application/json

{
  "username": "zhangsan",
  "password": "123456",
  "re_password": "123456"
}
```

## 3. Validator v10 标签详解

### 3.1 常用校验标签

Validator 库提供了丰富的校验标签,以下是最常用的:

| 标签 | 含义 | 示例 | 说明 |
|------|------|------|------|
| **required** | 必填字段 | `binding:"required"` | 不能为零值(空字符串、0、false 等) |
| **min** | 最小值/最小长度 | `binding:"min=3"` | 字符串:最少3个字符<br>数字:最小值为3 |
| **max** | 最大值/最大长度 | `binding:"max=64"` | 字符串:最多64个字符<br>数字:最大值为64 |
| **len** | 固定长度 | `binding:"len=11"` | 字符串长度必须为11(如手机号) |
| **email** | 邮箱格式 | `binding:"email"` | 必须是合法的邮箱地址 |
| **url** | URL 格式 | `binding:"url"` | 必须是合法的 URL |
| **oneof** | 枚举值 | `binding:"oneof=1 0 -1"` | 值只能是 1、0 或 -1 其中之一 |
| **gt/gte** | 大于/大于等于 | `binding:"gt=0"` | 数字必须大于 0 |
| **lt/lte** | 小于/小于等于 | `binding:"lt=100"` | 数字必须小于 100 |
| **eqfield** | 等于另一个字段 | `binding:"eqfield=Password"` | 必须与 Password 字段值相同 |

### 3.2 组合使用标签

多个标签用逗号分隔:

```go
type SignUpRequest struct {
    // 用户名:必填,长度3-64个字符
    Username string `json:"username" binding:"required,min=3,max=64"`

    // 密码:必填,长度6-64个字符
    Password string `json:"password" binding:"required,min=6,max=64"`

    // 确认密码:必填,且必须与 Password 字段相同
    RePassword string `json:"re_password" binding:"required,eqfield=Password"`

    // 邮箱:可选,但如果填写则必须是邮箱格式
    Email string `json:"email" binding:"omitempty,email"`
}
```

**注意:**
- `required` 和 `omitempty` 不能同时使用
- `omitempty` 表示"如果为空则跳过后续校验,否则执行后续校验"

### 3.3 实战案例:投票参数校验

在 Bluebell 的投票功能中,我们使用了 `oneof` 标签:

```go
// internal/dto/request/post.go

type VoteRequest struct {
    // 帖子 ID:必填
    PostID    int64 `json:"post_id" binding:"required,string"`

    // 投票方向:必填,且只能是 1(赞成)、0(取消) 或 -1(反对)
    Direction int8  `json:"direction" binding:"required,oneof=1 0 -1"`
}
```

**为什么使用 `oneof`?**

投票方向有明确的业务含义:
- `1`: 点赞(赞成)
- `-1`: 点踩(反对)
- `0`: 取消投票

如果用户传了 `2` 或 `-5`,说明前端有 bug 或用户恶意篡改请求,必须拒绝。

**测试示例:**

```bash
# ✅ 合法请求
curl -X POST http://localhost:8080/api/v1/vote \
  -H "Content-Type: application/json" \
  -d '{"post_id": 1001, "direction": 1}'

# ❌ 非法请求:direction 不在允许范围内
curl -X POST http://localhost:8080/api/v1/vote \
  -H "Content-Type: application/json" \
  -d '{"post_id": 1001, "direction": 5}'
# 响应: {"code": 1003, "msg": "direction必须是[1 0 -1]中的一个"}
```

## 4. Controller 层参数校验实现

### 4.1 标准校验流程

在 Bluebell 项目中,所有 Controller 的参数校验都遵循统一模式:

```go
// internal/handler/user.go

func SignUpHandler(c *gin.Context) {
    // 1️⃣ 创建参数结构体
    p := new(request.SignUpRequest)

    // 2️⃣ 参数绑定和校验
    if err := c.ShouldBindJSON(p); err != nil {
        // 3️⃣ 类型断言:判断是否是 Validator 校验错误
        errs, ok := err.(validator.ValidationErrors)
        if !ok {
            // 非校验错误(如 JSON 格式错误),返回通用错误
            ResponseError(c, CodeInvalidParam)
            return
        }

        // 4️⃣ 翻译错误信息为中文,并去除结构体名前缀
        ResponseErrorWithMsg(c, CodeInvalidParam, removeTopStruct(errs.Translate(trans)))
        return
    }

    // 5️⃣ 参数校验通过,调用业务逻辑
    if err := logic.SignUp(p); err != nil {
        // 业务错误处理
        if errors.Is(err, mysql.ErrorUserExist) {
            ResponseError(c, CodeUserExist)
            return
        }
        ResponseError(c, CodeServerBusy)
        return
    }

    // 6️⃣ 返回成功响应
    ResponseSuccess(c, nil)
}
```

### 4.2 为什么需要类型断言?

`ShouldBindJSON` 可能返回两类错误:

```go
// 错误类型1: JSON 解析错误
{
  "username": "zhangsan",
  "password": "123456",  // ← 少了逗号
  "re_password": "123456"
}
// 返回错误: invalid character '"' after object key:value pair

// 错误类型2: Validator 校验错误
{
  "username": "",        // ← 空字符串,违反 required 规则
  "password": "123456",
  "re_password": "123456"
}
// 返回错误: validator.ValidationErrors 类型
```

**处理逻辑:**

```go
errs, ok := err.(validator.ValidationErrors)
if !ok {
    // 不是 ValidationErrors,说明是 JSON 格式错误等问题
    ResponseError(c, CodeInvalidParam)
    return
}
// 是 ValidationErrors,需要翻译成用户友好的错误信息
ResponseErrorWithMsg(c, CodeInvalidParam, removeTopStruct(errs.Translate(trans)))
```

### 4.3 错误信息翻译

Validator 默认返回英文错误信息,且包含结构体名前缀:

```
ParamSignUp.Username: 'Username' 必须是字符串类型
ParamSignUp.Password: 'Password' 长度必须至少为6个字符
```

我们做了两层优化:

#### 第一层:国际化翻译

在 `controller/validator.go` 中初始化翻译器:

```go
import (
    "github.com/gin-gonic/gin/binding"
    "github.com/go-playground/validator/v10"
    zh_translations "github.com/go-playground/validator/v10/translations/zh"
    "github.com/go-playground/universal-translator"
)

var trans ut.Translator // 全局翻译器

func InitTrans(locale string) (err error) {
    if v, ok := binding.Validator.Engine().(*validator.Validate); ok {
        // 设置翻译器
        zhT := zh.New()  // 中文翻译器
        enT := en.New()  // 英文翻译器
        uni := ut.New(enT, zhT, enT)

        // 根据参数选择语言
        trans, ok = uni.GetTranslator(locale)
        if !ok {
            return fmt.Errorf("uni.GetTranslator(%s) failed", locale)
        }

        // 注册翻译器
        switch locale {
        case "zh":
            err = zh_translations.RegisterDefaultTranslations(v, trans)
        case "en":
            err = en_translations.RegisterDefaultTranslations(v, trans)
        default:
            err = en_translations.RegisterDefaultTranslations(v, trans)
        }
        return
    }
    return
}
```

**在 main.go 中调用:**

```go
// 初始化 Validator 翻译器
if err := controller.InitTrans("zh"); err != nil {
    fmt.Printf("init validator trans failed, err:%v\n", err)
    return
}
```

#### 第二层:去除结构体名前缀

翻译后的错误信息仍然包含 `ParamSignUp.` 前缀,用户体验不好。我们使用 `removeTopStruct` 函数去除:

```go
// controller/validator.go

func removeTopStruct(fields map[string]string) map[string]string {
    res := map[string]string{}
    for field, err := range fields {
        // 去掉结构体名前缀
        // 例如: "ParamSignUp.Username" → "Username"
        res[field[strings.Index(field, ".")+1:]] = err
    }
    return res
}
```

**效果对比:**

```go
// ❌ 优化前
{
  "code": 1003,
  "msg": "请求参数有误",
  "data": {
    "ParamSignUp.Username": "Username为必填字段",
    "ParamSignUp.Password": "Password长度必须至少为6个字符"
  }
}

// ✅ 优化后
{
  "code": 1003,
  "msg": "请求参数有误",
  "data": {
    "username": "username为必填字段",
    "password": "password长度必须至少为6个字符"
  }
}
```

**注意:** 优化后的字段名使用 JSON 标签(小写),而非 Go 字段名(大驼峰),这需要额外配置。

## 5. 自定义字段名显示

### 5.1 问题:字段名不友好

默认情况下,Validator 使用 Go 结构体的字段名(如 `Username`),但前端使用的是 JSON 字段名(如 `username`)。

**我们希望:**
- 错误信息中的字段名与前端 JSON 字段一致
- 便于前端解析和展示错误

### 5.2 解决方案:注册 TagNameFunc

在 `InitTrans` 函数中添加:

```go
func InitTrans(locale string) (err error) {
    if v, ok := binding.Validator.Engine().(*validator.Validate); ok {

        // 🔥 注册 JSON 标签名函数
        // 让 Validator 使用 json 标签作为字段名,而非 Go 字段名
        v.RegisterTagNameFunc(func(fld reflect.StructField) string {
            // 获取 json 标签的第一部分(逗号前)
            // 例如: `json:"username,omitempty"` → "username"
            name := strings.SplitN(fld.Tag.Get("json"), ",", 2)[0]
            if name == "-" {
                // json:"-" 表示忽略该字段
                return ""
            }
            return name
        })

        // ... 后续翻译器注册代码 ...
    }
    return
}
```

**效果:**

```go
type ParamSignUp struct {
    Username string `json:"username" binding:"required"`
}

// ❌ 未注册 TagNameFunc
errs.Translate(trans)
// 返回: {"Username": "Username为必填字段"}

// ✅ 注册 TagNameFunc 后
errs.Translate(trans)
// 返回: {"username": "username为必填字段"}
```

## 6. 跨字段校验

### 6.1 业务需求

注册时,用户需要输入两次密码以确保无误:

```json
{
  "username": "zhangsan",
  "password": "123456",
  "re_password": "123456"  // ← 必须与 password 相同
}
```

单个字段的 `binding` 标签无法实现这个逻辑,需要自定义校验。

### 6.2 方法1:使用 eqfield 标签

最简单的方式是使用内置的 `eqfield` 标签:

```go
type SignUpRequest struct {
    Username   string `json:"username" binding:"required"`
    Password   string `json:"password" binding:"required"`
    RePassword string `json:"re_password" binding:"required,eqfield=Password"`
    //                                               ^^^^^^^^^^^^^^^^^^^^^^
    //                                               必须等于 Password 字段
}
```

**优点:** 简单直接,一行代码搞定

**缺点:** 错误信息不友好

```json
{
  "code": 1003,
  "msg": "请求参数有误",
  "data": {
    "re_password": "re_password必须等于Password"  // ← 错误信息包含字段名,不够友好
  }
}
```

### 6.3 方法2:自定义结构体级别校验

为了更友好的错误提示,Bluebell 使用了结构体级别的自定义校验:

```go
// controller/validator.go

// SignUpParamStructLevelValidation 注册参数自定义校验函数
func SignUpParamStructLevelValidation(sl validator.StructLevel) {
    // 获取当前校验的结构体实例
    su := sl.Current().Interface().(models.ParamSignUp)

    // 业务逻辑:两次密码必须一致
    if su.Password != su.RePassword {
        // 报告错误:字段 re_password 不满足 eqfield 规则
        sl.ReportError(su.RePassword, "re_password", "RePassword", "eqfield", "password")
        //             ^^^^^^^^^^^^^^  ^^^^^^^^^^^^^  ^^^^^^^^^^^  ^^^^^^^^^  ^^^^^^^^^^
        //             字段值          JSON字段名     Go字段名      校验标签    标签参数
    }
}
```

**注册自定义校验:**

```go
func InitTrans(locale string) (err error) {
    if v, ok := binding.Validator.Engine().(*validator.Validate); ok {
        // ... 其他初始化代码 ...

        // 🔥 注册结构体级别校验
        v.RegisterStructValidation(SignUpParamStructLevelValidation, models.ParamSignUp{})
        //                          ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^  ^^^^^^^^^^^^^^^^^^^^^
        //                          自定义校验函数                     要校验的结构体类型
    }
    return
}
```

**优势:**
- 更灵活的校验逻辑(可以调用外部函数、查询数据库等)
- 可以自定义错误信息

## 7. 完整的参数校验示例

### 7.1 用户注册参数

```go
// internal/dto/request/user.go

type SignUpRequest struct {
    // 用户名:必填,3-64字符
    Username string `json:"username" binding:"required,min=3,max=64"`

    // 密码:必填,6-64字符
    Password string `json:"password" binding:"required,min=6,max=64"`

    // 确认密码:必填(通过自定义校验验证一致性)
    RePassword string `json:"re_password" binding:"required"`
}
```

### 7.2 用户登录参数

```go
// internal/dto/request/user.go

type LoginRequest struct {
    // 登录时只需要用户名和密码,不需要确认密码
    Username string `json:"username" binding:"required"`
    Password string `json:"password" binding:"required"`
}
```

**设计原则:** 注册和登录使用不同的参数结构体,虽然字段相似,但业务含义不同,解耦更灵活。

### 7.3 帖子列表查询参数

```go
// internal/dto/request/post.go

type PostListRequest struct {
    // 页码:默认为1
    Page int64 `json:"page" form:"page"`

    // 每页数量:默认为10
    Size int64 `json:"size" form:"size"`

    // 排序方式:"time" 或 "score"
    Order string `json:"order" form:"order"`

    // 社区ID:可选,按社区筛选
    CommunityID int64 `json:"community_id" form:"community_id"`
}
```

**注意:** `form:"page"` 标签允许从 URL 查询参数中绑定:

```bash
GET /api/v1/posts?page=2&size=20&order=score&community_id=1
```

## 8. 最佳实践

### 8.1 参数校验清单

在编写 Controller 时,确保做到:

- ✅ 所有必填字段都有 `required` 标签
- ✅ 字符串字段限制最大长度(防止超长攻击)
- ✅ 数字字段限制范围(防止非法值)
- ✅ 枚举字段使用 `oneof` 标签
- ✅ 邮箱、URL 等使用对应的格式校验标签
- ✅ 跨字段校验使用自定义校验函数
- ✅ 错误信息翻译成中文并去除结构体前缀

### 8.2 错误响应格式统一

```go
// ✅ 好的做法:使用统一的响应函数
if err := c.ShouldBindJSON(&p); err != nil {
    ResponseError(c, CodeInvalidParam)  // 统一格式
    return
}

// ❌ 不好的做法:直接返回 JSON
if err := c.ShouldBindJSON(&p); err != nil {
    c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})
    return
}
```

### 8.3 不要在 Controller 写业务逻辑

```go
// ❌ 错误示例:在 Controller 判断用户是否存在
func SignUpHandler(c *gin.Context) {
    var p request.SignUpRequest
    c.ShouldBindJSON(&p)

    // ❌ 这是业务逻辑,不应该在 Controller
    if mysql.CheckUserExist(p.Username) {
        ResponseError(c, CodeUserExist)
        return
    }
    // ...
}

// ✅ 正确示例:Controller 只负责参数和响应,业务逻辑交给 Logic 层
func SignUpHandler(c *gin.Context) {
    var p request.SignUpRequest
    if err := c.ShouldBindJSON(&p); err != nil {
        // Controller 只处理参数校验
        ResponseError(c, CodeInvalidParam)
        return
    }

    // ✅ 业务逻辑在 Logic 层
    if err := logic.SignUp(&p); err != nil {
        // Controller 只处理响应
        if errors.Is(err, mysql.ErrorUserExist) {
            ResponseError(c, CodeUserExist)
            return
        }
        ResponseError(c, CodeServerBusy)
        return
    }

    ResponseSuccess(c, nil)
}
```

### 8.4 参数命名规范

```go
// ✅ 好的命名
type SignUpRequest struct {
    Username   string `json:"username" binding:"required"`
    Password   string `json:"password" binding:"required"`
    RePassword string `json:"re_password" binding:"required"`
}

// ❌ 不好的命名
type SignUpRequest struct {  // 不要用 Request/Response 后缀
    Name string `json:"name"`  // 太模糊,应该是 Username
    Pwd  string `json:"pwd"`   // 缩写不清晰,应该是 Password
    Pwd2 string `json:"pwd2"`  // 语义不明确,应该是 RePassword
}
```

## 9. 常见问题 FAQ

### Q1: binding:"required" 和 binding:"omitempty" 有什么区别?

**A:**

- `required`: 字段不能为零值(空字符串、0、false、nil 等),**必须有值**
- `omitempty`: 字段为零值时**跳过后续校验**,非零值时执行后续校验

```go
// 示例1: Email 必填
type User struct {
    Email string `binding:"required,email"`  // 必须填写且格式正确
}

// 示例2: Email 可选
type User struct {
    Email string `binding:"omitempty,email"`  // 不填可以,填了就必须格式正确
}

// ❌ 错误用法
type User struct {
    Email string `binding:"required,omitempty,email"`  // 矛盾!不能同时使用
}
```

---

### Q2: 为什么要使用 ShouldBindJSON 而不是 BindJSON?

**A:**

| 方法 | 校验失败后 | 自定义错误响应 | 推荐场景 |
|------|-----------|---------------|---------|
| **ShouldBindJSON** | 返回 error,不自动响应 | ✅ 可以 | 生产环境,需要统一错误格式 |
| **BindJSON** | 自动返回 400 Bad Request | ❌ 不可以 | 快速原型开发 |

**示例:**

```go
// ✅ ShouldBindJSON:可控制响应格式
if err := c.ShouldBindJSON(&p); err != nil {
    ResponseError(c, CodeInvalidParam)  // 返回自定义错误码和格式
    return
}

// ❌ BindJSON:无法自定义
if err := c.BindJSON(&p); err != nil {
    // 这里永远不会执行,因为 BindJSON 已经自动响应了:
    // HTTP 400: {"error": "Key: 'ParamSignUp.Username' Error:Field validation..."}
    return
}
```

---

### Q3: 如何校验密码强度?

**A:**

Validator 没有内置密码强度校验,需要自定义:

```go
// 方法1:使用正则表达式(不推荐,正则难维护)
type ParamSignUp struct {
    // 密码必须包含大小写字母和数字
    Password string `binding:"required,min=8,regexp=^(?=.*[a-z])(?=.*[A-Z])(?=.*\\d).+$"`
}

// 方法2:自定义校验函数(推荐)
func validatePassword(fl validator.FieldLevel) bool {
    password := fl.Field().String()

    // 密码策略:至少8位,包含大小写字母和数字
    if len(password) < 8 {
        return false
    }

    hasUpper := false
    hasLower := false
    hasDigit := false

    for _, c := range password {
        if unicode.IsUpper(c) {
            hasUpper = true
        } else if unicode.IsLower(c) {
            hasLower = true
        } else if unicode.IsDigit(c) {
            hasDigit = true
        }
    }

    return hasUpper && hasLower && hasDigit
}

// 在 InitTrans 中注册
v.RegisterValidation("strong_password", validatePassword)

// 第04章结尾的自定义注册请求参数
type SignUpRequest struct {
    Password string `binding:"required,strong_password"`
}
```

---

### Q4: 如何处理可选字段?

**A:**

使用 `omitempty` 标签:

```go
type ParamSignUp struct {
    Username string `json:"username" binding:"required"`
    Password string `json:"password" binding:"required"`

    // Email 是可选的,但如果提供了,必须是合法的邮箱格式
    Email *string `json:"email" binding:"omitempty,email"`
    //    ^^^^^^^ 使用指针类型,区分"未提供"和"空字符串"
}
```

**区别:**

```go
// 情况1: 未提供 email 字段
{"username": "zhangsan", "password": "123456"}
// Email 为 nil,校验通过 ✅

// 情况2: 提供了 email,但为空
{"username": "zhangsan", "password": "123456", "email": ""}
// Email 为 "",校验失败 ❌ (空字符串不是合法邮箱)

// 情况3: 提供了合法 email
{"username": "zhangsan", "password": "123456", "email": "test@example.com"}
// Email 为 "test@example.com",校验通过 ✅
```

### Q5: 前端如何展示校验错误?

**A:**

后端返回的错误格式:

```json
{
  "code": 1003,
  "msg": "请求参数有误",
  "data": {
    "username": "username长度必须至少为3个字符",
    "password": "password长度必须至少为6个字符"
  }
}
```

前端可以这样处理:

```javascript
// Vue 3 示例
async function handleSignup() {
  try {
    await api.signup(formData)
  } catch (error) {
    if (error.response.data.code === 1003) {
      const errors = error.response.data.data

      // 方法1:逐个展示
      for (const [field, msg] of Object.entries(errors)) {
        ElMessage.error(`${field}: ${msg}`)
      }

      // 方法2:设置到表单校验提示
      Object.keys(errors).forEach(field => {
        formRef.value.setFieldError(field, errors[field])
      })
    }
  }
}
```

## 10. 知识回顾

通过本章学习,你应该掌握了:

✅ **参数校验的重要性**
   - 用户输入不可信,后端必须做完整校验
   - 多层校验策略:前端(体验) + 后端(安全) + 数据库(最后防线)

✅ **Gin 参数绑定**
   - `ShouldBindJSON` 从 JSON 请求体绑定参数
   - 绑定失败返回 error,可自定义错误响应
   - 参数结构体使用 `json` 和 `binding` 标签

✅ **Validator v10 标签**
   - `required`: 必填
   - `min/max`: 最小/最大长度或值
   - `email/url`: 格式校验
   - `oneof`: 枚举值校验
   - `eqfield`: 字段相等校验

✅ **错误信息优化**
   - 国际化翻译(中文/英文)
   - 去除结构体名前缀
   - 使用 JSON 字段名而非 Go 字段名

✅ **自定义校验**
   - `RegisterTagNameFunc`: 自定义字段名显示
   - `RegisterStructValidation`: 结构体级别跨字段校验
   - `RegisterValidation`: 注册自定义校验标签

✅ **最佳实践**
   - Controller 只负责参数校验和响应,不写业务逻辑
   - 使用统一的 `ResponseError` 和 `ResponseSuccess` 函数
   - 参数结构体定义在 `models/params.go` 中

type ParamChangePassword struct {
    OldPassword     string `json:"old_password" binding:"required"`
    NewPassword     string `json:"new_password" binding:"required,min=6,max=64"`
    ReNewPassword   string `json:"re_new_password" binding:"required"`

func ChangePasswordStructLevelValidation(sl validator.StructLevel) {

    // 新密码必须与旧密码不同
