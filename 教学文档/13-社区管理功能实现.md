## 1. 需求分析

## 2. 数据库设计

### 2.1 community 表结构

虽然项目中没有SQL文件,但根据代码可以推断表结构:

```sql
CREATE TABLE `community` (
  `id` int(11) NOT NULL AUTO_INCREMENT COMMENT '自增主键',
  `community_id` int(10) unsigned NOT NULL COMMENT '社区ID',
  `community_name` varchar(128) NOT NULL COMMENT '社区名称',
  `introduction` varchar(256) NOT NULL DEFAULT '' COMMENT '社区介绍',
  `create_time` timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time` timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `idx_community_id` (`community_id`),
  UNIQUE KEY `idx_community_name` (`community_name`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_general_ci COMMENT='社区表';
```

**字段说明:**

| 字段 | 类型 | 说明 | 为什么这样设计? |
|------|------|------|----------------|
| `id` | int(11) | 自增主键 | 内部使用,保证唯一性 |
| `community_id` | int(10) unsigned | 社区业务ID | 暴露给前端,可以控制ID生成规则 |
| `community_name` | varchar(128) | 社区名称 | 唯一索引,防止重复 |
| `introduction` | varchar(256) | 社区介绍 | 简短描述,首页不需要加载 |
| `create_time` | timestamp | 创建时间 | 记录社区建立时间 |
| `update_time` | timestamp | 更新时间 | 自动更新,方便追踪变更 |

**设计要点:**

**Q1: 为什么有两个ID(`id`和`community_id`)?**

A:
- `id`: 数据库自增主键,仅内部使用
- `community_id`: 业务ID,暴露给前端
- **好处:**
  - 可以自定义ID生成规则(如使用Snowflake)
  - 数据库迁移时可以保持业务ID不变
  - 前端看不到真实的数据库主键

**Q2: 为什么`community_name`需要唯一索引?**

A:
- 防止创建重名社区
- 加速按名称查询(`WHERE community_name = ?`)
- 保证业务逻辑正确性

## 3. API 设计

### 3.1 RESTful 规范

**RESTful** 是一种API设计风格,核心思想:**资源 + HTTP方法**

```
资源(名词)      HTTP方法(动词)     操作
/community      GET              获取社区列表
/community/:id  GET              获取社区详情
/community      POST             创建社区
/community/:id  PUT              更新社区
/community/:id  DELETE           删除社区
```

**Bluebell 社区接口设计:**

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 获取社区列表 | GET | `/api/v1/community` | 返回所有社区 |
| 获取社区详情 | GET | `/api/v1/community/:id` | 返回指定社区详情 |

## 4. 数据模型定义

### 4.1 internal/model/community.go

```go
package model

import "time"

// Community 社区结构体
// 使用 GORM ORM 映射数据库 community 表
type Community struct {
	// gorm:"column:community_id;primaryKey" 指定数据库列名为 community_id 并作为主键
	CommunityID  int64     `json:"id,string" gorm:"column:community_id;primaryKey"`
	Name         string    `json:"name" gorm:"column:community_name;uniqueIndex;size:128;not null"`
	Introduction string    `json:"introduction" gorm:"column:introduction;type:text"`
	CreateTime   time.Time `json:"create_time" gorm:"column:create_time;autoCreateTime"`
}

// TableName 自定义表名
// GORM 默认使用复数形式表名(communities),需要显式指定为 community
func (Community) TableName() string {
	return "community"
}
```

**字段说明:**

| 字段 | 类型 | JSON标签 | GORM标签 | 说明 |
|------|------|----------|----------|------|
| `CommunityID` | int64 | `id,string` | `column:community_id;primaryKey` | 社区ID,JSON序列化为字符串 |
| `Name` | string | `name` | `column:community_name;uniqueIndex` | 社区名称,唯一索引 |
| `Introduction` | string | `introduction` | `column:introduction;type:text` | 社区介绍 |
| `CreateTime` | time.Time | `create_time` | `column:create_time;autoCreateTime` | 创建时间,自动填充 |

---

### 4.2 ,string 标签详解

**问题:** JavaScript的Number类型精度问题

```javascript
// JavaScript 中
console.log(9007199254740992 === 9007199254740993);  // true (精度丢失!)
```

**解决:** Go的int64序列化为JSON字符串

```go
// ❌ 不加 ,string
type Community struct {
	ID int64 `json:"id"`  // → {"id": 1234567890123456789}
}

// ✅ 加 ,string
type Community struct {
	ID int64 `json:"id,string"`  // → {"id": "1234567890123456789"}
}
```

**前端处理:**
```javascript
// Vue/React 中直接当字符串使用
const communityId = data.id;  // "1"
axios.get(`/community/${communityId}`);  // /community/1
```

## 5. DAO层实现

### 5.1 internal/dao/mysql/community.go

```go
package mysql

import (
	"bluebell/internal/model"
	"context"

	"go.uber.org/zap"
	"gorm.io/gorm"
)

// CommunityRepo 社区数据访问实现
type CommunityRepo struct {
	db *gorm.DB // 注入的数据库连接
}

// NewCommunityRepo 创建 CommunityRepo 实例
func NewCommunityRepo(db *gorm.DB) *CommunityRepo {
	return &CommunityRepo{db: db}
}

// GetCommunityList 查询社区列表数据
func (r *CommunityRepo) GetCommunityList(ctx context.Context) (data []*model.Community, err error) {
	// 初始化切片,防止查询为空时返回 nil
	data = make([]*model.Community, 0)

	// 使用 GORM 查询,WithContext 传递上下文
	err = r.db.WithContext(ctx).Select("community_id", "community_name").Find(&data).Error
	if err != nil {
		zap.L().Error("query community list failed", zap.Error(err))
		return nil, err
	}

	return
}

// GetCommunityDetailByID 根据ID查询数据库详情
func (r *CommunityRepo) GetCommunityDetailByID(ctx context.Context, id int64) (community *model.Community, err error) {
	community = new(model.Community)

	// 使用 GORM 查询单条记录
	err = r.db.WithContext(ctx).
		Where("community_id = ?", id).
		First(community).Error

	if err != nil {
		// 特殊处理:如果没有查到数据
		if err == gorm.ErrRecordNotFound {
			zap.L().Warn("there is no community in db", zap.Int64("community_id", id))
			return nil, nil // 返回nil而不是error,让上层决定如何处理
		}
		// 其他数据库错误
		zap.L().Error("query community detail failed", zap.Error(err))
		return nil, err
	}
	return
}
```

---

### 5.2 核心要点解析

#### 5.2.1 GORM Find vs First

| 方法 | 用途 | 返回值 | ErrRecordNotFound |
|------|------|--------|-----------|
| `db.Find(&slice)` | 查询多行数据 | `[]T` | **不返回**ErrRecordNotFound(返回空切片) |
| `db.First(&struct)` | 查询单行数据 | `T` | **返回**ErrRecordNotFound |

**示例:**
```go
// Find: 查询0行不报错
var communities []*model.Community
err := r.db.WithContext(ctx).Find(&communities).Error
// err = nil, communities = []  ← 空切片

// First: 查询0行报错
var community model.Community
err := r.db.WithContext(ctx).Where("id = ?", 9999).First(&community).Error
// err = gorm.ErrRecordNotFound  ← 需要处理
```

---

#### 5.2.2 gorm.ErrRecordNotFound 处理策略

**场景1: 列表查询(GetCommunityList)**

```go
if err == gorm.ErrRecordNotFound {
    zap.L().Warn("there is no community in db")
    // ← GORM Find 不会返回此错误,返回空切片
}
```

**理由:**
- 列表为空是正常情况(如初始化状态)
- 前端期望得到`{"code": 1000, "data": []}`而不是错误

---

**场景2: 详情查询(GetCommunityDetailByID)**

```go
if err == gorm.ErrRecordNotFound {
    zap.L().Warn("there is no community in db", zap.Int64("community_id", id))
    return nil, nil  // ← 返回nil,不返回error
}
```

**理由:**
- 查不到数据可能是业务错误(无效ID),也可能是数据被删除
- 让上层(Logic层)决定如何处理
- 如果直接返回error,上层无法区分"数据不存在"和"数据库故障"

**对比:**
```go
// ❌ 不好的做法
if err == sql.ErrNoRows {
    return nil, err  // 上层无法区分错误类型
}

// ✅ 好的做法
if err == sql.ErrNoRows {
    return nil, nil  // 上层可以判断 data == nil 来区分
}
```

---

#### 5.2.3 为什么列表只返回部分字段?

```go
sqlStr := "select community_id, community_name from community"
```

**原因:**
1. **性能优化**: 不需要的字段不查询(如`introduction`很长)
2. **前端需求**: 列表页只需要ID和名称
3. **减少流量**: 返回的JSON更小

**对比:**
```sql
-- ❌ 查询所有字段(浪费)
SELECT * FROM community;

-- ✅ 只查询需要的字段
SELECT community_id, community_name FROM community;
```

## 6. Service层实现

### 6.1 internal/service/community/community_service.go

```go
package community

import (
	"bluebell/internal/domain/repository"
	"bluebell/internal/model"
	"bluebell/pkg/errorx"
	"context"
)

// CommunityService 社区业务逻辑服务
type CommunityService struct {
	communityRepo repository.CommunityRepository // 通过 Repository 接口访问数据
}

// NewCommunityService 创建社区服务实例 (构造函数,用于依赖注入)
func NewCommunityService(communityRepo repository.CommunityRepository) *CommunityService {
	return &CommunityService{communityRepo: communityRepo}
}

// GetCommunityList 获取社区列表
func (s *CommunityService) GetCommunityList(ctx context.Context) ([]*model.Community, error) {
	// 通过 Repository 接口调用 DAO 层
	return s.communityRepo.GetCommunityList(ctx)
}

// GetCommunityDetail 根据ID获取社区详情
func (s *CommunityService) GetCommunityDetail(ctx context.Context, id int64) (*model.Community, error) {
	// 通过 Repository 接口查询
	data, err := s.communityRepo.GetCommunityDetailByID(ctx, id)
	if err != nil {
		return nil, errorx.ErrServerBusy
	}
	if data == nil {
		return nil, errorx.ErrNotFound
	}
	return data, nil
}
```

## 7. Handler层实现

### 7.1 internal/handler/community_handler.go

```go
package handler

import (
	"bluebell/pkg/errorx"
	"strconv"

	"github.com/gin-gonic/gin"
	"go.uber.org/zap"
)

// CommunityHandler 获取社区列表
func (h *Handlers) CommunityHandler(c *gin.Context) {
	// 1. 查询所有的社区 (通过依赖注入调用 Service 层)
	data, err := h.Services.Community.GetCommunityList(c.Request.Context())
	if err != nil {
		zap.L().Error("CommunityService.GetCommunityList() failed", zap.Error(err))
		HandleError(c, errorx.ErrServerBusy)
		return
	}

	// 2. 返回成功响应
	ResponseSuccess(c, data)
}

// CommunityHandlerByID 获取社区详情
func (h *Handlers) CommunityHandlerByID(c *gin.Context) {
	// 1. 获取路径参数
	idStr := c.Param("id")

	// 2. 参数类型转换
	id, err := strconv.ParseInt(idStr, 10, 64)
	if err != nil {
		HandleError(c, errorx.ErrInvalidParam)
		return
	}

	// 3. 调用 Service 层获取详情 (通过依赖注入)
	data, err := h.Services.Community.GetCommunityDetail(c.Request.Context(), id)
	if err != nil {
		// 使用 HandleError 统一处理错误
		HandleError(c, err)
		return
	}

	// 4. 返回成功响应
	ResponseSuccess(c, data)
}
```

#### 7.2.1 路径参数解析

---

#### 7.2.2 strconv.ParseInt 详解

```go
id, err := strconv.ParseInt(idStr, 10, 64)
```

**参数说明:**
- `idStr`: 要转换的字符串
- `10`: 进制(10进制)
- `64`: 返回值位数(int64)

**返回值:**
- `id`: 转换后的int64
- `err`: 转换错误(如果输入不是数字)

**测试用例:**
```go
// ✅ 成功
strconv.ParseInt("123", 10, 64)    // 123, nil
strconv.ParseInt("0", 10, 64)      // 0, nil
strconv.ParseInt("-456", 10, 64)   // -456, nil

// ❌ 失败
strconv.ParseInt("abc", 10, 64)    // 0, error
strconv.ParseInt("12.34", 10, 64)  // 0, error
strconv.ParseInt("", 10, 64)       // 0, error
```

---

#### 7.2.3 错误处理策略

**区分两类错误:**

**1. 参数错误 (400 Bad Request)**
```go
if err != nil {
	// 转换失败 → 参数格式错误
	ResponseError(c, CodeInvalidParam)
	return
}

if errors.Is(err, errno.ErrorInvalidID) {
	// 业务错误 → 社区ID不存在
	ResponseError(c, CodeInvalidParam)
	return
}
```

**2. 系统错误 (500 Server Busy)**
```go
if err != nil {
	// 数据库连接失败、SQL语法错误等
	ResponseError(c, CodeServerBusy)
	return
}
```

## 10. 常见问题与解决方案

### 10.1 返回数据为null

**问题:**
```json
{
  "code": 1000,
  "msg": "success",
  "data": null  ← 为什么是null?
}
```

**原因1:** 数据库中没有数据

**解决:** 执行初始化SQL
```sql
INSERT INTO `community` VALUES
  ('1', '1', 'Go', 'Golang', NOW(), NOW());
```

---

**原因2:** 结构体标签错误

```go
// ❌ 错误
type Community struct {
	ID   int64  `json:"id" db:"id"`  // db标签错误
	Name string `json:"name" db:"name"`
}

// ✅ 正确
type Community struct {
	ID   int64  `json:"id" db:"community_id"`  // 对应数据库字段名
	Name string `json:"name" db:"community_name"`
}
```

---

### 10.2 路径参数获取失败

**问题:** `c.Param("id")` 返回空字符串

**原因:** 路由定义错误

```go
// ❌ 错误
r.GET("/community/id", handler)  // 没有 :

// ✅ 正确
r.GET("/community/:id", handler)  // 有 :
```

---

### 10.3 Token认证失败

**问题:** 返回401或1004错误

**原因1:** 忘记携带Token

**解决:**
```bash
# ❌ 错误

# ✅ 正确
curl -H "Authorization: Bearer <token>" http://localhost:8080/api/v1/community
```

---

**原因2:** Token格式错误

```bash
# ❌ 错误
Authorization: eyJhbGc...  # 少了 Bearer

# ✅ 正确
Authorization: Bearer eyJhbGc...
```

### 10.4 Swagger注释不生效

### 练习3: 添加缓存优化

**任务:** 使用Redis缓存社区列表,减少数据库查询

**要求:**
1. 社区列表缓存5分钟
2. 缓存未命中时查询数据库
3. 查询数据库后写入缓存

**参考答案:**
```go
// dao/redis/community.go
func GetCommunityListCache() ([]*models.CommunityDetail, error) {
	key := "bluebell:community:list"
	data, err := rdb.Get(ctx, key).Result()
	if err != nil {
		return nil, err
	}

	var communities []*models.CommunityDetail
	err = json.Unmarshal([]byte(data), &communities)
	return communities, err
}

func SetCommunityListCache(data []*models.CommunityDetail) error {
	key := "bluebell:community:list"
	jsonData, err := json.Marshal(data)
	if err != nil {
		return err
	}
	return rdb.Set(ctx, key, jsonData, 5*time.Minute).Err()
}

// logic/community.go
func GetCommunityList() ([]*models.CommunityDetail, error) {
	// 1. 尝试从缓存获取
	data, err := redis.GetCommunityListCache()
	if err == nil {
		return data, nil
	}

	// 2. 缓存未命中,查询数据库
	data, err = mysql.GetCommunityList()
	if err != nil {
		return nil, err
	}

	// 3. 写入缓存
	go redis.SetCommunityListCache(data)  // 异步写入

	return data, nil
}
```

## 13. 本章总结

### 13.1 核心知识点

| 知识点 | 说明 |
|--------|------|
| **RESTful API** | 资源+HTTP方法,`/community`表示社区资源 |
| **路径参数** | `/community/:id`,通过`c.Param("id")`获取 |
| **三层架构** | Controller → Logic → DAO,职责明确 |
| **sql.ErrNoRows** | 列表查询忽略,详情查询返回nil |
| **,string标签** | 解决JavaScript大数字精度问题 |
| **错误区分** | 参数错误(400) vs 系统错误(500) |
| **Swagger注释** | `@Param id path int true`,路径参数用`path` |

### 13.2 数据流转

```
HTTP请求 → Gin路由
           ↓
      Controller (参数解析、类型转换)
           ↓
      Logic (业务逻辑、未来加缓存)
           ↓
      DAO (SQL查询)
           ↓
      MySQL数据库
```

### 13.3 为什么这样设计?

**Q: 为什么不在DAO层直接返回error?**

A: 为了让Logic层能区分"数据不存在"和"数据库故障"两种情况,采用不同的处理策略。

---

**Q: 为什么列表和详情用同一个结构体?**

A: 虽然列表只返回部分字段,但使用同一个结构体更灵活。JSON序列化时,未赋值的字段自动被忽略。

---

**Q: 为什么需要Logic层?**

A: 当前虽然简单,但未来会加入缓存、数据拼接、权限校验等复杂逻辑,Logic层是扩展的基础。

## 15. 常见面试题

**Q1: RESTful中GET和POST的区别是什么?**

**A:**
- **语义:** GET用于查询,POST用于创建
- **幂等性:** GET是幂等的(多次请求结果相同),POST不是
- **缓存:** GET可以被缓存,POST不能
- **参数:** GET参数在URL中,POST在Body中

---

**Q2: 为什么社区ID要用,string标签?**

**A:** JavaScript的Number类型最大安全整数是`2^53-1 = 9007199254740991`,Go的int64最大值是`2^63-1 = 9223372036854775807`,会导致精度丢失。使用`,string`将int64序列化为字符串避免此问题。

---

**Q3: gorm.ErrRecordNotFound 什么时候会触发?**

**A:** 使用`db.First()`, `db.Take()`, `db.Last()`查询单行数据时,如果查询结果为空,会返回`gorm.ErrRecordNotFound`。但使用`db.Find()`查询多行时,结果为空不会返回此错误,而是空切片。建议使用`errors.Is(err, gorm.ErrRecordNotFound)`来判断。

---

**Q4: 三层架构中,业务逻辑应该放在哪一层?**

**A:** **Logic层**。Controller层只负责参数解析和响应封装,DAO层只负责数据访问,业务逻辑(如缓存策略、权限校验、数据拼接)都应该在Logic层实现。

---

**Q5: 如何设计一个好的RESTful API?**

**A:**
1. **名词化:** 使用资源名词而非动词(`/users`而非`/getUsers`)
2. **HTTP方法:** 利用GET/POST/PUT/DELETE表示操作
3. **版本控制:** 使用`/api/v1`前缀
4. **统一响应:** 使用统一的响应格式(code, msg, data)
5. **错误处理:** 区分业务错误和系统错误
6. **文档完善:** 提供Swagger等文档
