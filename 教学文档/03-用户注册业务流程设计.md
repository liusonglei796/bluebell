## 1. DDD 分层架构设计模式

### 1.1 为什么需要分层?

**反面教材:** 所有逻辑写在一个函数里

```go
// ❌ 糟糕的设计
func SignUpHandler(c *gin.Context) {
    // 1. 获取参数
    var p ParamSignUp
    c.ShouldBindJSON(&p)

    // 2. 参数校验
    if len(p.Username) < 3 { ... }

    // 3. 查询数据库
    db.QueryRow("SELECT * FROM user WHERE username=?", p.Username)

    // 4. 生成ID
    userID := snowflake.GenID()

    // 5. 密码加密
    hashedPwd, _ := bcrypt.GenerateFromPassword(...)

    // 6. 插入数据库
    db.Exec("INSERT INTO user ...", userID, p.Username, hashedPwd)

    // 7. 返回响应
    c.JSON(200, gin.H{"msg": "ok"})
}
```

**问题:**
- ❌ 函数过长,难以阅读
- ❌ 职责不清,难以测试
- ❌ 代码重复,难以复用
- ❌ 修改困难,容易出错

### 1.2 DDD 分层架构的设计

```
┌─────────────────────────────────────────────────────┐
│                    HTTP 请求                         │
│                       ↓                              │
│  ┌────────────────────────────────────────────┐    │
│  │  Handler 层 (请求处理层 - 交通警察)        │    │
│  │  职责: 参数解析、校验、调用业务、返回响应   │    │
│  └────────────────────────────────────────────┘    │
│                       ↓                              │
│  ┌────────────────────────────────────────────┐    │
│  │  Service 层 (业务逻辑层 - 大脑)            │    │
│  │  职责: 编排业务流程、调用 Repository 接口   │    │
│  └────────────────────────────────────────────┘    │
│                       ↓                              │
│  ┌────────────────────────────────────────────┐    │
│  │  Domain 层 (领域层 - 契约定义)             │    │
│  │  职责: 定义 Repository 接口和领域错误       │    │
│  └────────────────────────────────────────────┘    │
│                       ↓                              │
│  ┌────────────────────────────────────────────┐    │
│  │  DAO 层 (数据访问层 - 仓库管理员)          │    │
│  │  职责: 实现 Repository 接口、操作数据库     │    │
│  └────────────────────────────────────────────┘    │
│                       ↓                              │
│               Database (MySQL/Redis)                │
└─────────────────────────────────────────────────────┘
```

### 1.3 各层职责详解

#### Handler 层 (请求处理)

**职责:** "交通警察"

```go
// internal/handler/user_handler.go

func (h *Handlers) SignUpHandler(c *gin.Context) {
    // 1. 获取并校验参数
    p := new(request.SignUpRequest)
    if err := c.ShouldBindJSON(p); err != nil {
        HandleError(c, errorx.ErrInvalidParam)
        return
    }

    // 2. 调用业务逻辑 (通过依赖注入的 Services)
    if err := h.Services.User.SignUp(c.Request.Context(), p); err != nil {
        HandleError(c, err)
        return
    }

    // 3. 返回成功响应
    ResponseSuccess(c, nil)
}
```

**核心原则:**
- ✅ 只做参数处理和响应
- ✅ 不包含业务逻辑
- ✅ 不直接操作数据库
- ✅ 通过 `h.Services` 调用业务层 (依赖注入)

#### Service 层 (业务逻辑)

**职责:** "大脑" - 编排业务流程

```go
// internal/service/user/user_service.go

func (s *UserService) SignUp(ctx context.Context, p *request.SignUpRequest) error {
    // 1. 检查用户是否存在 (通过 Repository 接口)
    if err := s.userRepo.CheckUserExist(ctx, p.Username); err != nil {
        if errors.Is(err, repository.ErrUserExist) {
            return errorx.ErrUserExist
        }
        return errorx.ErrServerBusy
    }

    // 2. 生成用户ID (雪花算法)
    userID := snowflake.GetID()

    // 3. 构造用户对象
    u := &model.User{
        UserID:   userID,
        Username: p.Username,
        Password: p.Password, // 密码加密由 Model 的 BeforeCreate 钩子完成
    }

    // 4. 保存进数据库 (通过 Repository 接口)
    return s.userRepo.InsertUser(ctx, u)
}
```

**核心原则:**
- ✅ 实现核心业务逻辑
- ✅ 通过 Repository **接口** 访问数据层 (面向接口编程)
- ✅ 不关心数据如何存储 (MySQL/Redis 等)
- ✅ 返回明确的业务错误

#### DAO 层 (数据访问)

**职责:** "仓库管理员" - 实现 Repository 接口,执行数据操作

```go
// internal/dao/mysql/user.go

// CheckUserExist 检查用户是否存在
func (r *UserRepo) CheckUserExist(ctx context.Context, username string) error {
    var count int64
    err := r.db.WithContext(ctx).Model(&model.User{}).
        Where("username = ?", username).Count(&count).Error
    if err != nil {
        return err
    }
    if count > 0 {
        return repository.ErrUserExist // 返回 Domain 层定义的错误
    }
    return nil
}

// InsertUser 插入新用户
func (r *UserRepo) InsertUser(ctx context.Context, user *model.User) error {
    return r.db.WithContext(ctx).Create(user).Error
}
```

**核心原则:**
- ✅ 只负责数据库/Redis 操作
- ✅ 实现 Domain 层定义的 Repository 接口
- ✅ 返回 Domain 层定义的错误类型 (如 `repository.ErrUserExist`)
- ✅ 不包含业务逻辑

### 1.4 依赖注入与 Repository 接口

传统的三层架构中,Service 层直接调用 DAO 层的包函数。而 DDD 架构使用**依赖注入 + Repository 接口**,好处如下:

**传统方式 (紧耦合):**
```go
// ❌ Service 直接依赖具体的 DAO 包
func SignUp(p *models.ParamSignUp) error {
    mysql.CheckUserExist(p.Username)  // 直接调用包函数
    mysql.InsertUser(user)            // 如果换成 PostgreSQL,需要改这里
}
```

**DDD 方式 (松耦合):**
```go
// ✅ Service 只依赖 Repository 接口
type UserService struct {
    userRepo repository.UserRepository // 接口,不关心具体实现
}

func (s *UserService) SignUp(ctx context.Context, p *request.SignUpRequest) error {
    s.userRepo.CheckUserExist(ctx, p.Username)  // 通过接口调用
    s.userRepo.InsertUser(ctx, u)               // 换数据库只需改 DAO 层实现
}
```

**Repository 接口定义 (Domain 层):**
```go
// internal/domain/repository/mysql.go

type UserRepository interface {
    CheckUserExist(ctx context.Context, username string) error
    InsertUser(ctx context.Context, user *model.User) error
    CheckLogin(ctx context.Context, user *model.User) error
    GetUserByID(ctx context.Context, uid int64) (*model.User, error)
    GetUsersByIDs(ctx context.Context, ids []int64) ([]*model.User, error)
}
```

## 2. 注册业务完整流程

### 2.1 流程图

```
用户提交注册表单
    ↓
┌──────────────────────────────────┐
│  Handler: SignUpHandler          │
│  1. 解析 JSON 参数                │
│  2. 参数校验(validator)           │
└──────────────────────────────────┘
    ↓
┌──────────────────────────────────┐
│  Service: UserService.SignUp     │
│  1. 检查用户名是否已存在          │
│  2. 生成用户ID(Snowflake)        │
│  3. 构造 User 对象               │
│  4. 保存用户到数据库              │
│  (密码加密由 Model 钩子自动完成)  │
└──────────────────────────────────┘
    ↓
┌──────────────────────────────────┐
│  DAO: UserRepo                   │
│  1. CheckUserExist               │
│  2. InsertUser (触发 BeforeCreate)│
└──────────────────────────────────┘
    ↓
    返回成功/失败响应
```

### 2.2 数据流转


## 3. 代码实现

### 3.1 定义请求参数

#### 文件: `internal/dto/request/user.go`

```go
package request

// SignUpRequest 注册请求参数
type SignUpRequest struct {
	// binding 标签用于参数校验
	Username   string `json:"username" binding:"required"`
	Password   string `json:"password" binding:"required"`
	RePassword string `json:"re_password" binding:"required,eqfield=Password"`
}
```

**binding 标签说明:**

| 标签 | 含义 | 示例 |
|------|------|------|
| `required` | 必填字段 | `binding:"required"` |
| `eqfield` | 等于某个字段 | `binding:"eqfield=Password"` |
| `min` | 最小长度 | `binding:"min=6"` |
| `max` | 最大长度 | `binding:"max=20"` |

详细内容见 [第04章: 请求参数绑定与校验](./04-请求参数绑定与校验.md)

### 3.2 Handler 层实现

#### 文件: `internal/handler/user_handler.go`

```go
package handler

import (
	"bluebell/internal/dto/request"
	"bluebell/pkg/errorx"

	"github.com/gin-gonic/gin"
	"github.com/go-playground/validator/v10"
)

// SignUpHandler 处理用户注册请求
func (h *Handlers) SignUpHandler(c *gin.Context) {
	// 1. 参数校验
	p := new(request.SignUpRequest)
	if err := c.ShouldBindJSON(p); err != nil {
		// 获取validator.ValidationErrors类型的errors
		errs, ok := err.(validator.ValidationErrors)
		if !ok {
			// 非validator.ValidationErrors类型错误直接返回
			HandleError(c, errorx.ErrInvalidParam)
			return
		}
		// validator.ValidationErrors类型错误则进行翻译
		ResponseErrorWithMsg(c, errorx.CodeInvalidParam, removeTopStruct(errs.Translate(trans)))
		return
	}

	// 2. 业务处理 (通过依赖注入调用 Service 层)
	if err := h.Services.User.SignUp(c.Request.Context(), p); err != nil {
		// 3. 错误处理: 使用 HandleError 统一处理
		HandleError(c, err)
		return
	}

	// 4. 返回响应
	ResponseSuccess(c, nil)
}
```

**关键点解析:**

1. **Handlers 结构体与依赖注入**
```go
// Handler 层持有 Services 引用,通过依赖注入获得
type Handlers struct {
    Services *service.Services
}

// 调用方式: h.Services.User.SignUp(...)
// 而不是: logic.SignUp(...)
```

2. **统一错误处理**
```go
// HandleError 会自动识别 errorx.CodeError 并返回对应的错误码
// Service 层返回的错误已经包含了正确的业务错误码
HandleError(c, err)
```

3. **Context 传递**
```go
// 将 Gin 的 HTTP Context 传递给 Service 层
// 用于超时控制、请求追踪等
h.Services.User.SignUp(c.Request.Context(), p)
```

### 3.3 Service 层实现

#### 文件: `internal/service/user/user_service.go`

```go
package user

import (
	"bluebell/internal/domain/repository"
	"bluebell/internal/dto/request"
	"bluebell/internal/model"
	"bluebell/pkg/errorx"
	"bluebell/pkg/snowflake"
	"context"
	"errors"

	"go.uber.org/zap"
)

// UserService 用户业务逻辑服务
type UserService struct {
	userRepo   repository.UserRepository           // 用户数据 Repository (接口)
	tokenCache repository.UserTokenCacheRepository  // Token 缓存 Repository (接口)
}

// NewUserService 创建用户服务实例 (构造函数,用于依赖注入)
func NewUserService(userRepo repository.UserRepository, tokenCache repository.UserTokenCacheRepository) *UserService {
	return &UserService{
		userRepo:   userRepo,
		tokenCache: tokenCache,
	}
}

// SignUp 处理用户注册业务逻辑
func (s *UserService) SignUp(ctx context.Context, p *request.SignUpRequest) (err error) {
	// 1. 判断用户是否存在 (通过 Repository 接口)
	if err = s.userRepo.CheckUserExist(ctx, p.Username); err != nil {
		if errors.Is(err, repository.ErrUserExist) {
			return errorx.ErrUserExist // 转换为业务错误码
		}
		zap.L().Error("userRepo.CheckUserExist failed",
			zap.String("username", p.Username),
			zap.Error(err))
		return errorx.ErrServerBusy
	}

	// 2. 生成 UID (使用雪花算法)
	userID := snowflake.GetID()

	// 3. 构造 User 实例
	// 注意: 密码加密由 Model 的 BeforeCreate 钩子自动完成
	u := &model.User{
		UserID:   userID,
		Username: p.Username,
		Password: p.Password,
	}

	// 4. 保存进数据库 (通过 Repository 接口)
	err = s.userRepo.InsertUser(ctx, u)
	if err != nil {
		zap.L().Error("userRepo.InsertUser failed",
			zap.Int64("user_id", userID),
			zap.String("username", p.Username),
			zap.Error(err))
		return errorx.ErrServerBusy
	}

	return nil
}
```

**业务流程详解:**

**步骤1: 检查用户是否存在**
```go
if err = s.userRepo.CheckUserExist(ctx, p.Username); err != nil {
    // 通过 Repository 接口调用,不关心底层是 MySQL 还是 PostgreSQL
    // Domain 层定义的错误 repository.ErrUserExist 统一了错误类型
    if errors.Is(err, repository.ErrUserExist) {
        return errorx.ErrUserExist // 转换为带错误码的业务错误
    }
}
```

**步骤2: 生成用户ID**
```go
userID := snowflake.GetID()
// 生成的ID示例: 256294141623799808
// 全局唯一、趋势递增、不可预测
```

**步骤3: 密码加密 (Model 层自动处理)**
```go
// 密码加密逻辑在 model.User 的 BeforeCreate 钩子中
// Service 层不需要关心加密方式 — 这是 Model 层的职责
u := &model.User{
    Password: p.Password, // 传入明文密码
}
// InsertUser 触发 GORM Create → BeforeCreate 钩子自动加密
```

详见 [第06章: 用户密码加密与数据持久化](./06-用户密码加密与数据持久化.md)

### 3.4 DAO 层实现

#### 文件: `internal/dao/mysql/user.go`

```go
package mysql

import (
	"bluebell/internal/domain/repository"
	"bluebell/internal/model"
	"context"
	"fmt"

	"gorm.io/gorm"
)

// UserRepo 用户数据访问实现
type UserRepo struct {
	db *gorm.DB // 注入的数据库连接
}

// NewUserRepo 创建 UserRepo 实例
func NewUserRepo(db *gorm.DB) *UserRepo {
	return &UserRepo{db: db}
}

// CheckUserExist 检查用户是否存在
func (r *UserRepo) CheckUserExist(ctx context.Context, username string) (err error) {
	var count int64
	// GORM 使用 Count 方法统计记录数, WithContext 传递上下文
	err = r.db.WithContext(ctx).Model(&model.User{}).Where("username = ?", username).Count(&count).Error
	if err != nil {
		return err
	}
	if count > 0 {
		return repository.ErrUserExist // 返回 Domain 层定义的错误
	}
	return nil
}

// InsertUser 插入新用户
// 密码加密已移至 Model 的 BeforeCreate 钩子中自动处理
func (r *UserRepo) InsertUser(ctx context.Context, user *model.User) (err error) {
	err = r.db.WithContext(ctx).Create(user).Error
	if err != nil {
		return fmt.Errorf("插入用户失败: %w", err)
	}
	return nil
}
```

**关键设计:**

1. **结构体方法 vs 包级函数**
```go
// ❌ 旧方式: 包级函数,难以测试
func CheckUserExist(username string) error { ... }

// ✅ 新方式: 结构体方法,支持依赖注入
func (r *UserRepo) CheckUserExist(ctx context.Context, username string) error { ... }
```

2. **Domain 层错误**
```go
// 错误定义在 internal/domain/repository/errors.go 中
// Service 层通过 errors.Is() 判断,不依赖 DAO 包
var (
    ErrUserExist       = errors.New("用户已存在")
    ErrUserNotExist    = errors.New("用户不存在")
    ErrInvalidPassword = errors.New("密码错误")
)
```

3. **Context 传递**
```go
// WithContext 将请求上下文传递给数据库操作
// 支持超时取消、请求追踪等特性
r.db.WithContext(ctx).Model(...)
```

```go
package service

// Services 聚合所有 Service 实例
type Services struct {
```

## 6. 完整的请求响应流程

### 6.1 成功场景

```bash
# 请求
POST /api/v1/signup HTTP/1.1
Content-Type: application/json

{
    "username": "alice",
    "password": "123456",
    "re_password": "123456"
}

# 响应
HTTP/1.1 200 OK
Content-Type: application/json

{
    "code": 1000,
    "msg": "success",
    "data": null
}
```

**流程追踪:**
```
1. Gin 接收 HTTP 请求
2. 路由匹配到 h.SignUpHandler
3. Handler 解析并校验参数 ✅
4. 调用 h.Services.User.SignUp()
5. Service 检查用户不存在 ✅
6. Service 生成 ID: 256294141623799808
7. DAO 插入数据库 (Model.BeforeCreate 自动加密密码) ✅
8. Handler 返回成功响应
```

### 6.2 失败场景: 用户已存在

```bash
# 请求
POST /api/v1/signup
{
    "username": "alice",  # 已存在的用户名
    "password": "123456",
    "re_password": "123456"
}

# 响应
{
    "code": 1002,
    "msg": "用户名已存在",
    "data": null
}
```

**流程追踪:**
```
1-4. (同上)
5. DAO 返回 repository.ErrUserExist
6. Service 识别错误,返回 errorx.ErrUserExist (code=1002)
7. Handler 调用 HandleError(c, err)
8. HandleError 自动提取 CodeError 中的错误码和消息
```

### 6.3 失败场景: 参数错误

```bash
# 请求
{
    "username": "al",  # 太短
    "password": "123456",
    "re_password": "654321"  # 不匹配
}

# 响应
{
    "code": 1001,
    "msg": "请求参数错误",

}
```

**流程追踪:**
```
1-2. (同上)
3. Handler 参数校验失败 ❌
4. 调用 HandleError(c, errorx.ErrInvalidParam)
(不会执行 Service 层)
```

## 7. 错误码设计

#### 文件: `pkg/errorx/errorx.go`

```go
package errorx

// CodeError 带业务错误码的自定义错误
type CodeError struct {
	Code  int    // 业务错误码
	Msg   string // 错误消息
	cause error  // 被包装的底层错误
}

// 业务错误码常量定义
const (
	CodeSuccess      = 1000 // 成功
	CodeInvalidParam = 1001 // 请求参数错误
	CodeUserExist    = 1002 // 用户已存在
	CodeUserNotExist = 1003 // 用户不存在
	CodeServerBusy   = 1005 // 服务繁忙
	// ... 更多错误码
)

// 预定义常用错误实例
var (
	ErrInvalidParam = New(CodeInvalidParam, "请求参数错误")
	ErrUserExist    = New(CodeUserExist, "用户名已存在")
	ErrServerBusy   = New(CodeServerBusy, "服务繁忙")
)
```

**与旧方式的对比:**
```go
// ❌ 旧方式: controller/code.go 中的 ResCode + codeMsgMap
type ResCode int64
const CodeUserExist ResCode = 1003

// ✅ 新方式: errorx.CodeError 结构体,错误码和消息绑定在一起
var ErrUserExist = errorx.New(CodeUserExist, "用户名已存在")

// Handler 层无需判断错误类型,直接透传
HandleError(c, err) // 自动提取 code 和 msg
```

## 8. 知识回顾与总结

### 8.1 DDD 分层架构总结

```
┌──────────────────────────────────────────────┐
│        DDD 分层 + 依赖注入原则               │
├──────────────────────────────────────────────┤
│  Handler: 我只做交警                         │
│    - 接收请求、校验参数、返回响应             │
│    - 通过 h.Services 调用业务层              │
│                                              │
│  Service: 我是大脑                           │
│    - 编排业务流程                             │
│    - 通过 Repository 接口 访问数据层         │
│                                              │
│  Domain: 我是契约                            │
│    - 定义 Repository 接口                    │
│    - 定义领域错误                             │
│                                              │
│  DAO: 我是仓库管理员                          │
│    - 实现 Repository 接口                    │
│    - 不关心业务逻辑                           │
└──────────────────────────────────────────────┘
```

### 8.2 注册流程核心步骤

1. ✅ 参数校验(Handler)
2. ✅ 检查用户是否存在(Service → Repository)
3. ✅ 生成用户ID(Snowflake)
4. ✅ 密码加密(Model.BeforeCreate 钩子)
5. ✅ 保存数据库(Service → Repository → DAO)
