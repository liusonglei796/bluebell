## 📖 章节内容详解

### 1. 问题回顾

展示了改造前存在的问题：
- 所有方法混在 Handlers 中
- 依赖具体实现而非接口
- 难以进行单元测试
- 违反 SOLID 原则

### 2. 解决方案

介绍了三个核心改进：
- **拆分 Handler**：将 Handlers 拆分为 userHandler、postHandler 等
- **构造函数注入**：每个 Handler 通过构造函数接收依赖
- **接口依赖**：Handler 只依赖 Service 接口，不依赖实现

### 3. DI 容器设计

详细设计了 HandlerProvider：
```go
type HandlerProvider struct {
    UserHandler      *userHandler
    PostHandler      *postHandler
    CommunityHandler *communityHandler
    VoteHandler      *voteHandler
}
```

5. 注册路由

## 💡 关键知识点

### 1. 依赖注入的三种方式

- **构造函数注入** ✅ (本章重点)
- 属性注入
- 方法参数注入

### 2. 为什么要依赖接口？

```go
// ❌ 错误：依赖实现
type Handler struct {
    service *service.UserService
}

// ✅ 正确：依赖接口
type Handler struct {
    service domain.service.UserService
}
```

**原因**：
- 解耦代码
- 支持 Mock 测试
- 易于扩展

### 3. DI 容器的作用

- 统一管理依赖
- 清晰展示依赖关系
- 便于组织和维护

### 4. 单元测试的价值

通过 Mock 注入进行隔离测试，提高代码质量。

### 2026-03-06 更新说明

#### 1. 实现 Unit of Work 模式并修复 NewRepositories 未定义问题
- **背景**：原有的 Handler DI 改造文档引用了 `mysql.NewRepositories`，但代码中缺少具体实现，导致主程序无法编译。
- **改进**：
  - 在 `internal/dao/mysql/` 下新增了 `repositories.go` 文件。
  - 实现了 `repositoriesUOWStruct`，满足 `repointerface.UnitOfWork` 接口。
  - 提供了 `NewRepositories(db *gorm.DB)` 构造函数，支持通过 GORM 实例初始化所有仓储，并支持数据库事务。
- **意义**：完善了项目的 DI 架构链条（Infrastructure -> DAO -> Service -> Handler），使数据访问层也能被统一管理，并为 Service 层提供了一致的事务处理能力。

#### 2. DAO 层错误处理统一使用 errorx.Wrap
- 将 `internal/dao/mysql/` 下所有 DAO 层的错误处理从 `fmt.Errorf` 改为 `errorx.Wrap`
- 统一使用 `errorx.CodeDBError` 作为数据库错误的错误码
- 涉及的错误信息：查询用户失败、插入用户失败、登录失败、查询帖子失败、创建帖子失败、删除帖子失败、查询社区列表失败、查询社区详情失败等
- 确保错误能够携带业务错误码传播到上层

#### 2. Service 层错误检查改用 errorx.GetCode
- 将 `internal/service/` 下所有 Service 层的错误检查从 `errors.Is` 改为 `errorx.GetCode`
- 原因：`errorx.Wrap` 包装后的错误使用 `errors.Is` 无法正确匹配，需要通过错误码判断
- 涉及的修改：
  - `user_service.go`：`CheckUserExist` 错误检查改用 `errorx.GetCode(err) == errorx.CodeUserExist`
  - `user_service.go`：`CheckLogin` 错误检查改用 `errorx.GetCode(err)` 判断 `CodeUserNotExist` 和 `CodeInvalidPassword`
  - `vote_service.go`：`VoteForPost` 错误检查改用 `errorx.GetCode(err)` 判断 `CodeVoteTimeExpire` 和 `CodeVoteRepeated`

#### 3. 删除未使用的 import
- 移除 `errors` 包在 Service 层的使用（DAO 层仍需使用 `errors.Is` 检查 `gorm.ErrRecordNotFound`）

#### 4. 错误处理最佳实践总结
- **DAO 层**：使用 `errorx.Wrap(err, errorx.CodeDBError, "具体操作描述")` 包装底层错误
- **Service 层**：使用 `errorx.GetCode(err) == errorx.CodeXxx` 判断具体业务错误
- **Handler 层**：直接返回 Service 层错误，由统一错误处理中间件处理
- **关键点**：确保错误码能够从 DAO 层一直传递到 Handler 层

#### 1. Handler 层结构优化
- 将 `userHandler`、`postHandler` 等结构体重命名为 `userHandlerStruct`、`postHandlerStruct` 等，明确其为结构体类型，符合项目内部约定的命名规范。
- 外部包（如 `router`）通过 `HandlerProvider` 中导出的字段（如 `UserHandler`）访问这些结构体。

#### 2. Handler 处理流程标准化
- 统一了所有 Handler 方法的内部逻辑，严格遵循以下 5 步流程：
    1. **参数绑定**：使用 `ShouldBindJSON`/`ShouldBindQuery` 等绑定数据。
    2. **验证错误处理**：使用 `errors.As` 显式判断 `validator.ValidationErrors` 并翻译。
    3. **通用错误处理**：捕获其他类型的绑定错误。
    4. **业务逻辑调用**：通过注入的 Service 接口执行业务，并统一处理 Service 层返回的错误。
    5. **成功响应**：统一使用 `backfront.ResponseSuccess` 返回结果。
- 在代码中增加了详细的步骤注释（1. 到 5.），提高了代码的可读性和规范性。

#### 3. 新增全局请求超时配置
- 在配置文件中引入了 `timeout.timeout` 配置项。
- 路由初始化时会自动解析该配置，并通过 `middleware.TimeoutMiddleware` 全局应用。
- 增加了对持续时间（Duration）字符串的解析校验，使用 `errorx` 统一报错。

#### 3. 限流配置优化
- 修复了 `RateLimit.FillInterval` 解析时的错误处理逻辑，统一使用 `errorx` 包装错误，提高了系统初始化的健壮性。

#### 6. DAO 层结构优化
- 参照 Handler 和 Service 层的规范，将 MySQL 和 Redis 的 DAO 实现类（如 `UserRepo`、`VoteCache`）重命名为 `userRepoStruct`、`voteCacheStruct` 等含有 `Struct` 后缀的私有形式。
- 这一改进确保了全栈架构（Handler -> Service -> DAO）在命名和可见性控制上的一致性，严格遵循面向接口编程的原则。
