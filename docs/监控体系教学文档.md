## 1. 什么是可观测性？

想象你开了一家餐厅，你需要知道：

| 问题 | 对应手段 | 类比 |
|------|---------|------|
| 今天来了多少客人？ | **指标（Metrics）** | 门口计数器 |
| 客人投诉菜太咸，是哪道菜？ | **日志（Logging）** | 厨房记事本 |
| 客人从进门到离开经历了什么？ | **追踪（Tracing）** | 全程摄像头 |

这就是**可观测性三支柱**：Metrics + Logging + Tracing。

## 2. 项目监控三大支柱

Bluebell 项目使用了以下技术栈：

| 支柱 | 技术 | 作用 |
|------|------|------|
| **日志** | `zap` + `lumberjack` | 记录运行事件，支持文件轮转 |
| **追踪** | `OpenTelemetry` + `Jaeger` | 追踪一个请求从进到出的完整路径 |
| **指标** | `Prometheus` + `Grafana` | 统计 QPS、延迟、业务数据等 |

## 3. 第一部分：日志（Logging）

### 3.1 什么是日志？

日志就是程序的"记事本"，记录发生了什么：

```
[INFO] 2024-01-01 用户张三登录成功
[ERROR] 2024-01-01 数据库连接超时
[WARN]  2024-01-01 帖子审核不通过，已隐藏
```

### 3.2 项目中的日志初始化

**文件：** `internal/infrastructure/logger/logger.go`

```go
func Init(cfg *config.Config, mode string) error {
    // 1. 创建文件写入器（支持自动轮转）
    writeSyncer := getWriteSyncer(cfg.Log.FileName, cfg.Log.MaxSize, cfg.Log.MaxAge, cfg.Log.MaxBackups)
    
    // 2. 根据运行模式配置不同级别
    if mode == "dev" {
        // 开发模式：同时输出到文件 + 控制台，Debug 级别
        filecore := zapcore.NewCore(JSON编码器, writeSyncer, Debug级别)
        consolecore := zapcore.NewCore(控制台编码器, os.Stdout, Debug级别)
        core = zapcore.NewTee(filecore, consolecore) // Tee = 同时写两份
    } else {
        // 生产模式：只写文件，Error 级别（减少噪音）
        core = zapcore.NewCore(JSON编码器, writeSyncer, Error级别)
    }
    
    // 3. 创建 logger 并注册为全局
    logger := zap.New(core, zap.AddCaller()) // AddCaller = 记录日志时带上文件名和行号
    zap.ReplaceGlobals(logger)
    return nil
}
```

**在 main.go 中调用：**
```go
// 第 58 行：初始化日志
logger.Init(cfg, cfg.App.Mode)
defer zap.L().Sync() // 程序退出时刷新缓冲区，确保日志不丢失
```

### 3.3 如何使用日志

在任意代码中：

```go
import "go.uber.org/zap"

// 记录普通信息
zap.L().Info("用户登录成功", zap.String("username", "张三"))

// 记录警告
zap.L().Warn("帖子审核不通过", zap.String("post_id", "123"))

// 记录错误（带错误对象）
zap.L().Error("数据库查询失败", zap.Error(err))

// 调试信息（仅开发模式可见）
zap.L().Debug("查询参数", zap.Any("params", p))
```

**输出示例（开发模式）：**
```
INFO    handler/post_handler/handler.go:84    用户发帖成功    {"user_id": 1, "post_id": "12345"}
```

### 3.4 日志轮转（lumberjack）

日志文件会越来越大，`lumberjack` 自动帮你管理：

```go
lumberjack.Logger{
    Filename:   "./logs/bluebell.log",  // 日志文件路径
    MaxSize:    100,                     // 单个文件最大 100MB
    MaxAge:     30,                      // 保留 30 天
    MaxBackups: 10,                      // 最多保留 10 个备份文件
}
```

效果：
```
bluebell.log           ← 当前正在写的
bluebell-20240101.log  ← 昨天的备份
bluebell-20231231.log  ← 前天的备份
...
```

## 4. 第二部分：链路追踪（Tracing）

### 4.1 什么是链路追踪？

一个 HTTP 请求进来后，可能经过：
```
用户请求 → 网关 → 认证 → Handler → Service → 数据库
                                    → Redis
                                    → MQ → AI审核
                                    → MQ → ES同步
```

**问题：** 如果请求慢了，是哪个环节慢了？

**答案：** 链路追踪给每个请求一个唯一的 `TraceID`，记录每一步的耗时。

### 4.2 项目中的 Tracing 初始化

**文件：** `internal/infrastructure/otel/tracer.go`

```go
func InitTracerProvider(cfg *config.Config) func() {
    // 1. 创建 OTLP 导出器（把追踪数据发送到 Jaeger）
    exporter, _ := otlptracegrpc.New(ctx,
        otlptracegrpc.WithEndpoint("http://localhost:4318"), // Jaeger 地址
        otlptracegrpc.WithInsecure(),
    )
    
    // 2. 创建资源标识（告诉 Jaeger "我是 bluebell 服务"）
    res, _ := resource.New(ctx,
        resource.WithAttributes(
            semconv.ServiceName("bluebell"),    // 服务名
            semconv.ServiceVersion("1.0.0"),    // 版本号
        ),
    )
    
    // 3. 创建 TracerProvider（追踪数据的总管家）
    tp := sdktrace.NewTracerProvider(
        sdktrace.WithBatcher(exporter),  // 批量发送，提高效率
        sdktrace.WithResource(res),
    )
    
    // 4. 注册为全局
    otel.SetTracerProvider(tp)
    
    // 5. 返回 shutdown 函数（程序退出时调用）
    return func() {
        tp.Shutdown(ctx) // 等待所有数据发送完毕
    }
}
```

**在 main.go 中调用：**
```go
// 第 85 行：初始化 OpenTelemetry
otelShutdown := otel.InitTracerProvider(cfg)
defer otelShutdown() // 程序退出时优雅关闭
```

### 4.4 手动追踪：自定义 Span

自动追踪只能记录 HTTP 层面的信息。要追踪业务逻辑，需要手动创建 Span。

#### Handler 层：StartSpan

```go
// handler/post_handler/handler.go
func (h *Handler) CreatePostHandler(c *gin.Context) {
    // 创建子 span，标识这是哪个 handler
    ctx, span := middleware.StartSpan(c, "bluebell/handler", "CreatePostHandler")
    defer span.End() // 函数返回时关闭 span
    
    // 业务逻辑...
    postID, err := h.postService.CreatePost(ctx, p, userID)
    if err != nil {
        middleware.RecordError(span, err) // 记录错误到 span
        backfront.HandleError(c, err)
        return
    }
    
    backfront.ResponseSuccess(c, nil)
}
```

#### Service 层：StartSpanFromContext

Service 层拿不到 `gin.Context`，所以用标准 `context.Context`：

```go
// service/postsvc/post_service.go
func (s *postServiceStruct) CreatePost(ctx context.Context, p *CreatePostRequest, authorID int64) (string, error) {
    // 从 context 创建子 span
    ctx, span := middleware.StartSpanFromContext(ctx, "bluebell/service", "CreatePost",
        attribute.Int64("user.id", authorID),
        attribute.Int64("community.id", p.CommunityID),
    )
    defer span.End()
    
    // 业务逻辑...
    postID := snowflake.GenID()
    err := s.postRepo.CreatePost(ctx, post)
    if err != nil {
        middleware.RecordError(span, err)
        return "", errorx.ErrServerBusy
    }
    
    return postID, nil
}
```

一个发帖请求的完整追踪：

```

│
├── GET /api/v1/post (otelgin 自动创建) ────── 总耗时 45ms
│     属性: http.method=POST, http.status=200
│
│     ├── CreatePostHandler (bluebell/handler) ── 40ms
│     │     属性: user.id=1
│     │

│     └──
└──
```

在 Jaeger UI 中可以看到这样的瀑布图，每一步的耗时一目了然。

### 4.6 RecordError：记录错误到 Span

```go
func RecordError(span trace.Span, err error) {
    if err == nil || span == nil {
        return
    }
    span.RecordError(err)              // 在 span 上记录错误事件
    span.SetStatus(codes.Error, err.Error()) // 标记 span 为错误状态
}
```

**使用场景：**
```go
if err := h.postService.CreatePost(ctx, p, userID); err != nil {
    middleware.RecordError(span, err) // ← 在 Jaeger 中这个 span 会标红
    backfront.HandleError(c, err)
    return
}
```

## 5. 第三部分：指标监控（Metrics）

### 5.1 什么是指标？

指标是**数字化的统计数据**，回答"有多少"、"多快"等问题：

| 指标 | 类型 | 含义 |
|------|------|------|
| `http_requests_total` | Counter（只增） | 总请求数 |
| `http_request_duration_seconds` | Histogram（分布） | 请求耗时分布 |
| `active_users` | Gauge（可增可减） | 当前在线人数 |
| `total_votes` | Counter | 总投票数 |
| `total_posts` | Counter | 总帖子数 |
| `audit_results` | CounterVec（带标签） | 审核结果统计 |

### 5.2 Prometheus 自动指标

**文件：** `internal/middleware/prometheus.go`

#### 5.2.1 定义指标收集器

```go
// 1. 请求总数（按方法、路径、状态码分类）
var httpRequestsTotal = prometheus.NewCounterVec(
    prometheus.CounterOpts{
        Name: "http_requests_total",
        Help: "Total number of HTTP requests",
    },
    []string{"method", "path", "status"}, // 三个维度
)

// 2. 请求耗时（按方法和路径分类）
var httpRequestDurationSeconds = prometheus.NewHistogramVec(
    prometheus.HistogramOpts{
        Name:    "http_request_duration_seconds",
        Help:    "HTTP request duration in seconds",
        Buckets: prometheus.DefBuckets, // 默认桶: 5ms, 10ms, 25ms, 50ms, 100ms...
    },
    []string{"method", "path"},
)
```

#### 5.2.2 注册指标

```go
func RegisterPrometheusMetrics() {
    prometheus.Register(httpRequestsTotal)
    prometheus.Register(httpRequestDurationSeconds)
}
```

#### 5.2.3 中间件自动采集

```go
func PrometheusMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        start := time.Now() // 记录开始时间
        
        c.Next() // 执行后续 handler
        
        // 请求完成后记录指标
        status := c.Writer.Status()      // 200, 404, 500...
        method := c.Request.Method       // GET, POST...
        path := c.FullPath()             // /api/v1/post/:id（不是原始 URL）
        
        // 请求数 +1
        httpRequestsTotal.WithLabelValues(method, path, statusToCode(status)).Inc()
        
        // 记录耗时
        httpRequestDurationSeconds.WithLabelValues(method, path).Observe(time.Since(start).Seconds())
    }
}
```

**在路由中注册（router.go）：**
```go
r.Use(
    middleware.PrometheusMiddleware(), // ← 自动记录每个请求的指标
)

// 暴露 /metrics 端点供 Prometheus 抓取
r.GET("/metrics", middleware.PrometheusHandler())
```

## 6. 完整调用链路图解

### 6.1 监控数据流向

```
┌─────────────────────────────────────────────────────────────────┐
│                        Bluebell 应用                             │
│                                                                 │
│  请求进入 → TracingMiddleware ──→ PrometheusMiddleware           │
│                ↓                        ↓                       │
│          创建 Span                记录指标数据                    │
│                ↓                        ↓                       │
│          Handler (StartSpan)      Counter.Inc()                 │
│                ↓                        ↓                       │
│          Service (StartSpanFromCtx) Histogram.Observe()         │
│                ↓                                                │
│          zap.L().Info/Error/Warn                                │
│                ↓                                                │
└─────────────────────────────────────────────────────────────────┘
                ↓                    ↓                    ↓
        ┌───────┴──────┐    ┌───────┴──────┐    ┌───────┴──────┐
        │   OTLP       │    │   /metrics   │    │   日志文件    │
        │   Exporter   │    │   Endpoint   │    │   ./logs/    │
        └───────┬──────┘    └───────┬──────┘    └───────┬──────┘
                ↓                   ↓                   ↓
        ┌───────┴──────┐    ┌───────┴──────┐    ┌───────┴──────┐
        │    Jaeger    │    │  Prometheus  │    │  Lumberjack  │
        │   (追踪UI)    │    │  (指标存储)   │    │  (日志轮转)   │
        └──────────────┘    └───────┬──────┘    └──────────────┘
                                    ↓
                            ┌───────┴──────┐
                            │   Grafana    │
                            │  (可视化看板)  │
                            └──────────────┘
```


## 7. 如何查看监控数据

### 7.1 查看日志

```bash
# 实时查看日志
tail -f ./logs/bluebell.log

# 搜索错误日志
grep "ERROR" ./logs/bluebell.log

# 搜索特定帖子
grep "post_id.*12345" ./logs/bluebell.log
```

### 7.2 查看链路追踪（Jaeger）

1. 启动 Jaeger（Docker）：
```bash
docker run -d --name jaeger \
  -e COLLECTOR_OTLP_ENABLED=true \
  -p 16686:16686 \
  -p 4318:4318 \
  jaegertracing/all-in-one:latest
```

2. 访问 Jaeger UI：`http://localhost:16686`
3. 选择服务 `bluebell`，点击 "Find Traces"
4. 点击某个 Trace 查看完整的瀑布图

### 7.3 查看指标（Prometheus）

1. 启动 Prometheus（docker-compose.yml）：
```yaml
prometheus:
  image: prom/prometheus
  ports:
    - "9090:9090"
  volumes:
    - ./prometheus.yml:/etc/prometheus/prometheus.yml
```

2. `prometheus.yml` 配置：
```yaml
scrape_configs:
  - job_name: 'bluebell'
    static_configs:
      - targets: ['host.docker.internal:8080']
```

3. 访问 Prometheus UI：`http://localhost:9090`
4. 输入查询语句，如：
   - `http_requests_total` → 总请求数
   - `rate(http_requests_total[5m])` → 5 分钟内的 QPS
   - `histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))` → P95 延迟

### 7.4 可视化看板（Grafana）

1. 启动 Grafana：
```bash
docker run -d --name grafana -p 3000:3000 grafana/grafana
```

2. 访问 Grafana：`http://localhost:3000`（默认账号 admin/admin）
3. 添加 Prometheus 数据源：`http://host.docker.internal:9090`
4. 创建 Dashboard，添加面板：

| 面板名称 | PromQL 查询 | 图表类型 |
|---------|------------|---------|
| QPS | `rate(http_requests_total[1m])` | 折线图 |
| P95 延迟 | `histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[1m]))` | 折线图 |
| 错误率 | `rate(http_requests_total{status="5xx"}[1m]) / rate(http_requests_total[1m])` | 百分比图 |
| 总帖子数 | `total_posts` | 数字面板 |
| 审核通过率 | `audit_results{result="safe"} / (audit_results{result="safe"} + audit_results{result="violation"})` | 饼图 |
