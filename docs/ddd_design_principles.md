# Bluebell 项目 DDD 设计原则

本文档记录了 Bluebell 项目在重构过程中遵循的领域驱动设计（DDD）核心原则，旨在明确各层职责，提升代码的可维护性和业务内聚性。

## 1. 领域层 vs 应用层：逻辑分类原则

业务逻辑并不是全部写在领域层，而是根据其本质分为“领域逻辑”和“应用逻辑”。

### 1.1 领域层 (Domain Layer) —— 核心规则 (Core Rules)
**职责**：处理那些无论采用何种技术实现，都必须遵守的业务规则。
*   **例子**：
    *   **实体校验**：帖子标题不能为空、评论内容非空。
    *   **权限规则**：只有作者本人才能删除自己的帖子。
    *   **安全策略**：用户密码在持久化前必须加密。
    *   **业务计算**：帖子分数的计算公式、投票权重的逻辑。
*   **特点**：
    *   **不依赖技术细节**：不关心数据库（MySQL/Redis）、不关心通信协议（HTTP/gRPC）。
    *   **高内聚**：业务逻辑与数据结构（实体）紧密结合。

### 1.2 应用层 (Application Layer) —— 流程编排 (Orchestration)
**职责**：作为指挥官，协调领域层、仓储层和外部服务，完成一个完整的用例（Use Case）。
*   **典型流程**：
    1.  开启数据库事务。
    2.  调用仓储（Repository）加载领域实体。
    3.  **调用领域层方法**执行业务逻辑或状态转换。
    4.  调用仓储保存变更后的实体。
    5.  调用缓存层同步数据。
    6.  发送异步消息或邮件。
    7.  提交/回滚事务。
*   **特点**：
    *   **无业务规则**：它不判断“对错”，只负责“步骤”。
    *   **面向用例**：一个 Service 方法通常对应前端的一个接口请求。

## 2. 职责边界对比

| 维度 | 领域层 (Domain) | 应用层 (Application) |
| :--- | :--- | :--- |
| **形象理解** | **厨师** (钻研菜谱、掌握火候) | **服务员/经理** (接单、传菜、收钱) |
| **核心关注点** | 业务的正确性、完整性、安全性 | 业务的流程、事务、外部系统交互 |
| **测试方式** | 单元测试 (纯 Go 逻辑，无需 Mock 数据库) | 集成测试 (需要 Mock 仓储或连接真实数据库) |
| **修改频率** | 较低 (业务规则变更时修改) | 较高 (流程变化、技术选型变更时修改) |

## 3. 开发建议

在编写一段代码时，请自问：
> **“如果我明天把数据库从 MySQL 换成 MongoDB，或者把 Web 框架换掉，这段逻辑还需要改吗？”**

*   如果 **不需要改** $\rightarrow$ 它是领域逻辑，请放进 `internal/domain/entity` 或 `internal/domain/service`。
*   如果 **需要改** $\rightarrow$ 它多半是应用流程或基础设施细节，请放进 `internal/application` 或 `internal/infrastructure`。
