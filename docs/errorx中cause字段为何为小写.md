# 为什么 `CodeError` 中的 `cause` 字段是小写的？

在 Go 语言中，结构体字段的首字母大小写决定了它的**可见性（访问权限）**：

- **首字母大写（如 `Code`、`Msg`）**：表示这个字段是**导出的（公开的/Public）**。包外部的代码（如其他的 Controller 或 Service）可以直接通过 `.Code` 访问或修改它，并且它能被 `json.Marshal` 正常序列化输出给前端。
- **首字母小写（如 `cause`）**：表示这个字段是**未导出的（私有的/Private）**。它只能在当前包（即 `errorx` 包）内部被访问，外部包无法直接拿到或修改它，同时也**不能**被 JSON 序列化。

## 为什么把 `cause` 设为小写（私有）？

这是非常有意的设计，主要出于以下三个核心考量：

### 1. 信息安全过滤（防止底层堆栈泄露给前端）

当我们将一个 API 错误通过 JSON 返回给前端时，我们只想让前端看到对业务有用的业务码和提示信息，例如：

```json
{
    "code": 1008,
    "msg": "资源不存在"
}
```

如果将其设计为导出的 `Cause`，那么在 JSON 序列化时，原本包裹的底层错误（如数据库报出的包含表结构名称或账号密码的 SQL 语法异常，或者是系统内网 IP 和堆栈调用信息）会被连同其他字段一起，原封不动地返回给前端！这会造成极其严重的**系统内部细节泄露**。把 `cause` 维持私有，`json.Marshal` 在序列化结构体时就会**自动忽略它**。

### 2. 维持良好的封装性（遵循 Go 的 `errors` 官方规范）

对于外部调用方来说，其实不需要也**不应该**知道底层被包装了哪个具体的错误实例的私有状态。

如果别的包（比如上层的中间件或恢复机制）确实需要获取或者解包查看到底里面装了什么底层错误，Go 官方推荐的标准解包姿势是利用内置机制——让自定义错误类型实现 `Unwrap() error` 接口。

```go
// errorx 包内部实现了 Unwrap，让外部通过标准库安全地获取 cause
func (e *CodeError) Unwrap() error {
    return e.cause 
}
```

外部需要查验底层抛错原因时，不是通过暴力地去读写 `err.cause` 这种读取结构体私有属性的方式，而是优雅地调用 `errors.Unwrap(err)` 或利用 `errors.As`/`errors.Is` 交由 Go 官方底层机制去层层扒开包装。这也是标准库 `fmt.Errorf("%w")` 所采取的同款做法。

### 3. 保持状态的安全不可变性

一个 `CodeError` 当被成功创建并赋予了相关的原始错误和业务提示（通过 `errorx.Wrap` 构造等）后，在整个调用链路返回的过程中，它的状态应该是安全的，随时被外层业务私自篡改其核心异常链是一件非常危险的事情。

将此字段设置为未导出，则防患于未然：只有在 `errorx` 包内部的构建函数能在初始化时拼装它，外部一旦封装完成，只能读（通过 `Unwrap`），彻底断绝了写（篡改）的可能。
